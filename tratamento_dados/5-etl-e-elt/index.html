    5. ETL e ELT#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    ETL = Extract, Transform, Load
    Processo de integração em que os dados são:

    1. extraídos das fontes;
    2. transformados em uma área intermediária ou motor de processamento;
    3. carregados no destino final, tipicamente um data warehouse.

    ELT = Extract, Load, Transform
    Processo em que os dados são:

    1. extraídos das fontes;
    2. carregados primeiro no destino;
    3. transformados dentro do ambiente de destino, geralmente um banco analítico, data lakehouse ou warehouse com alto poder computacional.

    Como funciona internamente#

    ETL#

    • Extract: coleta dados de bancos relacionais, arquivos CSV/JSON/XML, APIs, logs, sistemas legados.
    • Staging area: área temporária para armazenamento bruto/intermediário.
    • Transform: limpeza, padronização, deduplicação, enriquecimento, junções, agregações, aplicação de regras de negócio.
    • Load: gravação no destino, podendo ser:
      • full load (carga completa);
      • incremental load (somente alterações).

    ELT#

    • Extração semelhante ao ETL.
    • Os dados são carregados rapidamente no destino, muitas vezes em formato bruto.
    • A transformação ocorre com SQL, procedures, views materializadas ou engines distribuídas no próprio ambiente analítico.

    Componentes principais#

    • Fontes de dados
    • Conectores/ingestão
    • Área de staging
    • Motor de transformação
    • Metadados
    • Orquestração/agendamento
    • Monitoramento e logs
    • Destino analítico: data warehouse, data mart, data lake, lakehouse

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    • ETL transforma antes de carregar.
    • ELT transforma depois de carregar.
    • ETL é tradicional em ambientes com warehouse clássico.
    • ELT é comum em cloud analytics e big data.
    • Staging area é frequente em ETL, mas também pode existir em ELT.
    • Carga pode ser full ou incremental.
    • Transformações incluem:
      • limpeza
      • padronização
      • validação
      • deduplicação
      • enriquecimento
      • agregação
      • conformidade semântica
    • ETL/ELT dependem de metadados e linhagem de dados.
    • ETL/ELT podem operar em batch ou quase em tempo real/streaming.
    • ETL não é sinônimo de data warehouse, mas é fortemente associado a ele.
    • ELT não elimina transformação; apenas muda o local onde ela ocorre.
    • CDC (Change Data Capture) é técnica importante para cargas incrementais.
    • Qualidade de dados é etapa central, não acessória.
    • ETL/ELT podem aplicar regras de negócio, mas excesso de regra no pipeline pode dificultar manutenção.

    Classificações relevantes#

    • Por periodicidade:
      • batch
      • near real time
      • streaming
    • Por estratégia de carga:
      • full load
      • incremental
      • append-only
      • upsert/merge
    • Por local da transformação:
      • fora do destino = ETL
      • no destino = ELT

    3. Regras Técnicas e Comportamento#

    Funcionamento detalhado#

    Extração#

    • Pode ser:
      • completa: lê toda a base;
      • incremental: lê apenas novos/alterados.
    • Técnicas:
      • timestamp de atualização;
      • coluna de versão;
      • logs de transação;
      • CDC.

    Transformação#

    Operações típicas:

    • conversão de tipos;
    • tratamento de nulos;
    • padronização de datas, moedas e códigos;
    • remoção de duplicidades;
    • validação referencial;
    • derivação de atributos;
    • conformação de dimensões;
    • cálculo de métricas.

    Carga#

    • Append: só insere.
    • Overwrite/replace: substitui conjunto anterior.
    • Upsert/Merge: insere ou atualiza.
    • Pode haver controle de:
      • integridade;
      • transação;
      • idempotência;
      • reprocessamento.

    Limitações#

    ETL#

    • Pode exigir infraestrutura intermediária robusta.
    • Transformação prévia pode aumentar latência.
    • Menos flexível para exploração de dados brutos.

    ELT#

    • Exige destino com grande capacidade computacional.
    • Pode carregar dados ainda “sujos”, exigindo governança forte.
    • Se mal projetado, aumenta custo de processamento no warehouse.

    Condições de uso#

    ETL tende a ser melhor quando:

    • há necessidade de forte saneamento antes da carga;
    • o destino tem menor capacidade de processamento;
    • deseja-se entregar dados já altamente estruturados.

    ELT tende a ser melhor quando:

    • o destino é escalável e analítico;
    • há grande volume de dados;
    • deseja-se preservar dados brutos para múltiplos usos;
    • o processamento SQL/distribuído no destino é vantajoso.

    4. Diferenças Críticas#

    ETL x ELT#

    AspectoETLELT
    OrdemExtrai, transforma, carregaExtrai, carrega, transforma
    Local da transformaçãoAntes do destinoNo destino
    Uso típicoDW tradicionalCloud DW, lakehouse, big data
    LatênciaPode ser maiorPode ser menor na ingestão
    Dados brutos no destinoNem sempreFrequentemente sim

    ETL/ELT x Data Integration#

    • Integração de dados é conceito mais amplo.
    • ETL/ELT são estratégias/processos dentro da integração de dados.

    ETL x Migração de dados#

    • Migração: transferência pontual entre sistemas.
    • ETL: processo recorrente de integração e preparação.

    ETL x Virtualização de dados#

    • ETL materializa dados no destino.
    • Virtualização acessa dados sem necessariamente movê-los.

    Confusão comum da banca#

    • Dizer que no ELT não há transformação: falso.
    • Dizer que ETL é sempre em lote: falso.
    • Dizer que ELT dispensa staging: falso, não necessariamente.

    5. Exemplos e Aplicações#

    Exemplo técnico de transformação SQL no ELT#

    INSERT INTO fato_vendas (data_id, produto_id, valor_total)
SELECT
    d.id AS data_id,
    p.id AS produto_id,
    SUM(v.valor) AS valor_total
FROM vendas_raw v
JOIN dim_data d ON d.data_calendario = v.data_venda
JOIN dim_produto p ON p.codigo_origem = v.cod_produto
WHERE v.status = 'CONFIRMADA'
GROUP BY d.id, p.id;

    Exemplo de regras típicas#

    • Converter “01/02/2025” para tipo DATE.
    • Padronizar UF para sigla maiúscula.
    • Eliminar registros duplicados por CPF.
    • Substituir valores nulos por padrão controlado.
    • Rejeitar registros inválidos para quarentena.

    Cenários reais#

    • ETL: órgão público consolida dados de folha, patrimônio e orçamento em DW institucional.
    • ELT: empresa carrega logs e transações em warehouse cloud e transforma com SQL analítico.
    • CDC + incremental: somente registros alterados desde a última execução são processados.

    6. Pegadinhas de Prova#

    • “ETL e ELT diferem apenas no nome.” Errado.
    • “No ELT, a transformação ocorre na origem.” Errado.
    • “ETL pressupõe limpeza e integração antes da carga.” Certo.
    • “ELT é inviável em ambientes modernos de nuvem.” Errado; é justamente muito comum.
    • “Carga incremental sempre substitui a carga completa.” Errado; depende do cenário.
    • “Staging é obrigatório em toda arquitetura ETL/ELT.” Errado; é comum, não obrigatório.
    • Termos absolutos invalidadores:
      • sempre
      • nunca
      • somente
      • exclusivamente

    ConceitoRegraPegadinha

    ELTcarrega antes e transforma no destinoafirmar que não existe transformação

    Data warehousedestino estruturado analíticoconfundir com o próprio ETL

    8. Simulado (Certo ou Errado)#

    1. No processo ETL, os dados são transformados antes de serem carregados no repositório de destino.
    2. No ELT, a transformação deixa de existir, pois os dados são utilizados em seu formato original.
    3. Carga incremental é técnica que busca processar apenas dados novos ou alterados desde a última execução.
    4. ETL é necessariamente um processo em lote, não podendo operar em fluxos próximos do tempo real.
    5. Em arquiteturas ELT, é comum aproveitar o poder computacional do próprio data warehouse para executar transformações.
    6. Staging area é componente conceitualmente possível em ETL, mas não é requisito absoluto em toda implementação.
    7. CDC é uma abordagem útil para identificar alterações em dados de origem e apoiar cargas incrementais.
    8. ETL e migração de dados são conceitos idênticos, pois ambos consistem em mover dados entre sistemas.

    Gabarito comentado#

    1. Certo. Essência do ETL: transformar antes da carga.
    2. Errado. Pegadinha clássica: no ELT a transformação existe, mas ocorre depois da carga.
    3. Certo. Define corretamente carga incremental.
    4. Errado. ETL pode existir em batch, near real time e até streaming.
    5. Certo. Essa é uma das marcas do ELT moderno.
    6. Certo. Staging é comum, mas não universalmente obrigatório.
    7. Certo. CDC é técnica típica para detectar mudanças.
    8. Errado. Migração costuma ser pontual; ETL é processo de integração/preparação, geralmente recorrente.
