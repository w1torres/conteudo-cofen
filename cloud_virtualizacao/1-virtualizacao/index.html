    1. Virtualização#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Virtualização é a abstração dos recursos físicos de computação para criar ambientes lógicos isolados, chamados máquinas virtuais (VMs), capazes de executar sistemas operacionais e aplicações como se fossem máquinas físicas independentes.

    Ela permite que um único hardware físico hospede múltiplos sistemas operacionais simultaneamente, com isolamento de CPU, memória, armazenamento e rede.

    Como funciona internamente#

    O elemento central é o hipervisor (Virtual Machine Monitor - VMM), responsável por:

    • intermediar o acesso das VMs ao hardware;
    • alocar e controlar recursos;
    • garantir isolamento entre VMs;
    • emular ou paravirtualizar dispositivos.

    Tipos de hipervisor#

    Tipo 1 (bare metal / nativo)
    Executa diretamente sobre o hardware, sem sistema operacional hospedeiro.
    Ex.: VMware ESXi, Microsoft Hyper-V Server/Hyper-V role sobre Windows com arquitetura de hipervisor tipo 1.

    Tipo 2 (hosted)
    Executa sobre um sistema operacional hospedeiro.
    Ex.: VMware Workstation, VirtualBox.

    Componentes principais#

    • Host: máquina física que fornece recursos.
    • Guest: sistema operacional convidado dentro da VM.
    • Hipervisor: camada de virtualização.
    • vCPU: processador virtual atribuído à VM.
    • vRAM: memória virtual.
    • vDisk: disco virtual (VMDK no VMware, VHD/VHDX no Hyper-V).
    • vNIC: interface de rede virtual.
    • Datastore: repositório de arquivos de VM.
    • Snapshot/Checkpoint: estado salvo da VM.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    • Virtualização ≠ computação em nuvem; é tecnologia habilitadora da nuvem.
    • Hipervisor tipo 1 executa diretamente no hardware.
    • Hipervisor tipo 2 depende de SO hospedeiro.
    • Virtualização pode ser de servidor, desktop, aplicação, rede, armazenamento.
    • Emulação ≠ virtualização.
    • Paravirtualização exige cooperação/modificação do guest ou drivers específicos.
    • Virtualização completa abstrai o hardware para o guest.
    • Snapshots não substituem backup.
    • Alta consolidação aumenta eficiência, mas pode gerar contenção de recursos.
    • Live migration permite mover VM em execução entre hosts.
    • HA (High Availability) ≠ FT (Fault Tolerance).
    • VMware e Hyper-V usam formatos e ferramentas próprias.
    • Hyper-V usa checkpoints; VMware usa snapshots.
    • VMware ESXi é hipervisor tipo 1.
    • Hyper-V é classificado em prova como hipervisor tipo 1.
    • Overcommit de memória pode existir, dependendo da plataforma/configuração.
    • Virtualização melhora uso de recursos, mas adiciona overhead.

    Classificações importantes#

    Quanto ao método

    • Virtualização completa
    • Paravirtualização
    • Virtualização assistida por hardware

    Quanto ao objeto

    • Servidores
    • Desktops (VDI)
    • Aplicações
    • Rede (NFV, switches virtuais)
    • Armazenamento

    Quanto à mobilidade

    • Cold migration: VM desligada
    • Live migration / vMotion: VM ligada

    3. Regras Técnicas e Comportamento#

    Funcionamento detalhado#

    A VM enxerga hardware virtual. O hipervisor traduz operações da VM para o hardware real. Em processadores modernos, recursos como Intel VT-x e AMD-V reduzem overhead e melhoram desempenho.

    CPU#

    O hipervisor agenda vCPUs sobre CPUs físicas.
    Pegadinha: atribuir mais vCPUs do que o necessário pode piorar desempenho por escalonamento.

    Memória#

    Cada VM recebe memória virtualizada. Técnicas:

    • ballooning;
    • memory overcommit;
    • paginação/swapping.

    Pegadinha: overcommit pode aumentar densidade, mas pode degradar fortemente o desempenho.

    Armazenamento#

    Discos virtuais são arquivos ou LUNs.
    Formatos:

    • VMware: VMDK
    • Hyper-V: VHD, VHDX (mais moderno e resiliente)

    Rede#

    As VMs conectam-se por switches virtuais. Modos comuns:

    • bridge/externo;
    • NAT;
    • host-only/interno/privado.

    Limitações#

    • overhead de virtualização;
    • contenção de recursos;
    • dependência de compatibilidade de hardware;
    • snapshots prolongados degradam desempenho;
    • nem toda carga crítica tolera alta consolidação.

    Condições de uso#

    Virtualização é ideal para:

    • consolidação de servidores;
    • testes e homologação;
    • recuperação de desastres;
    • ambientes isolados;
    • elasticidade operacional.

    4. Diferenças Críticas#

    Virtualização x Emulação#

    • Virtualização: usa hardware real subjacente, com baixa perda relativa.
    • Emulação: reproduz outra arquitetura/hardware, com maior overhead.

    Virtualização x Contêineres#

    • VM: virtualiza hardware; cada instância tem seu próprio SO.
    • Contêiner: virtualiza no nível do sistema operacional; compartilha kernel do host.

    Pegadinha clássica: contêiner não substitui integralmente VM em todos os cenários, especialmente quando se exige isolamento forte ou SOs distintos.

    Snapshot x Backup#

    • Snapshot/Checkpoint: captura estado para retorno rápido.
    • Backup: cópia protegida para restauração e retenção.

    Snapshot não é estratégia completa de proteção de dados.

    HA x FT#

    • HA: reinicia VM em outro host após falha.
    • FT: mantém cópia sincronizada para continuidade quase imediata.

    VMware x Hyper-V#

    VMware

    • Hipervisor principal: ESXi
    • Gerenciamento: vCenter Server
    • Migração a quente: vMotion
    • Alta disponibilidade: vSphere HA
    • Tolerância a falhas: FT
    • Disco virtual: VMDK

    Hyper-V

    • Integrado ao ecossistema Microsoft
    • Gerenciamento: Hyper-V Manager, Failover Cluster Manager, System Center
    • Migração a quente: Live Migration
    • Armazenamento: VHD/VHDX
    • Checkpoints em vez de snapshots (terminologia Microsoft)

    5. Exemplos e Aplicações#

    Cenários reais#

    1. Consolidação: 10 servidores físicos subutilizados viram 2 hosts com várias VMs.
    2. Homologação: criação rápida de VMs para testes de versões.
    3. Recuperação: replicação de VMs para site secundário.
    4. VDI: desktops virtuais centralizados.

    Exemplo técnico: modos de switch virtual no Hyper-V#

    • External: conecta VM à rede física.
    • Internal: comunicação entre host e VMs.
    • Private: comunicação apenas entre VMs.

    Exemplo técnico: comandos PowerShell no Hyper-V#

    Get-VM
Start-VM -Name "SRV-APP01"
Checkpoint-VM -Name "SRV-APP01"

    Exemplo conceitual VMware#

    • Host ESXi executa várias VMs.
    • vCenter centraliza administração.
    • vMotion move VM ligada entre hosts compatíveis.

    6. Pegadinhas de Prova#

    • “Hipervisor tipo 2 executa diretamente no hardware.” Errado.
    • “Snapshot é equivalente a backup.” Errado.
    • “Virtualização elimina totalmente o overhead.” Errado.
    • “Toda virtualização depende de emulação.” Errado.
    • “Contêineres executam sistemas operacionais completos independentes.” Errado, em regra compartilham o kernel.
    • “Hyper-V é hipervisor hospedado do tipo 2.” Errado para fins de prova.
    • “Live migration exige desligamento da VM.” Errado.
    • “Paravirtualização dispensa qualquer adaptação do guest.” Errado.

    Termos que invalidam questões#

    Palavras absolutas como:

    • sempre
    • nunca
    • exclusivamente
    • totalmente
    • apenas

    Em virtualização, muitas assertivas erram por excesso de generalização.

    8. Simulado (Certo ou Errado)#

    1. O hipervisor tipo 1 executa diretamente sobre o hardware físico, sem depender de sistema operacional hospedeiro.
    2. No Hyper-V, checkpoints são sinônimos perfeitos de backup corporativo de longo prazo.
    3. O VMware ESXi é exemplo clássico de hipervisor bare metal.
    4. A live migration permite mover uma máquina virtual em execução entre hosts, com mínima interrupção.
    5. Em virtualização, a emulação e a virtualização completa são conceitos idênticos.
    6. Contêineres e máquinas virtuais são equivalentes, pois ambos virtualizam hardware e executam kernel próprio.
    7. O formato VHDX está associado ao Hyper-V, enquanto VMDK está associado ao VMware.
    8. Alta disponibilidade e tolerância a falhas são termos rigorosamente idênticos em ambientes virtualizados.

    Gabarito comentado#

    1. Certo.
    Definição clássica de hipervisor tipo 1. Pegadinha: confundir com hosted.

    2. Errado.
    Checkpoint/snapshot não substitui backup. Serve para retorno rápido de estado.

    3. Certo.
    ESXi é hipervisor tipo 1 da VMware.

    4. Certo.
    Live migration/vMotion move VM ligada. Pegadinha: dizer que precisa desligar.

    5. Errado.
    Emulação reproduz arquitetura/hardware; virtualização usa abstração com acesso ao hardware real.

    6. Errado.
    Contêineres normalmente compartilham o kernel do host; VMs possuem SO convidado próprio.

    7. Certo.
    Associação correta e muito cobrada.

    8. Errado.
    HA reinicia serviço/VM após falha; FT mantém redundância sincronizada para continuidade mais imediata.
