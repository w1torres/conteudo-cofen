    1. Linguagem Qlik#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    “Linguagem Qlik” costuma aparecer em prova como o conjunto de recursos de script, expressões e modelagem associativa usados nas plataformas QlikView e Qlik Sense para carga, transformação, associação e análise de dados.

    Não é uma linguagem de programação de propósito geral como Java ou Python. É uma linguagem orientada à análise de dados, com dois eixos principais:

    • Script de carga (load script): usado em ETL/ELT leve, integração e modelagem.
    • Expressões: usadas em gráficos, KPIs, tabelas e objetos analíticos.

    Como funciona internamente#

    O núcleo do Qlik é o motor associativo. Em vez de depender exclusivamente de joins executados em tempo de consulta, o Qlik:

    • carrega dados em memória;
    • cria associações automáticas entre tabelas por nomes de campos idênticos;
    • permite navegação por seleções, destacando:
      • valores selecionados;
      • valores associados;
      • valores não associados.

    Isso difere do modelo tradicional puramente relacional de consulta SQL ad hoc.

    Componentes principais#

    • LOAD: comando central do script.
    • SELECT: usado para buscar dados de fontes externas via conectores/ODBC/OLE DB.
    • Resident Load: carrega a partir de tabela já carregada em memória.
    • Preceding Load: empilha transformações antes do carregamento final.
    • QVD: formato nativo de armazenamento do Qlik, otimizado para performance.
    • Set Analysis: técnica para filtrar contexto de cálculo em expressões.
    • Data model viewer: visualização do modelo associativo.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    • Qlik usa modelo associativo, não apenas relacional clássico.
    • Associação entre tabelas ocorre por campos com mesmo nome.
    • Campos com mesmo nome e significado diferente podem gerar associação indevida.
    • Synthetic keys surgem quando há múltiplos campos comuns entre tabelas.
    • LOAD é o comando mais característico do script Qlik.
    • SELECT busca dados da origem; LOAD transforma e estrutura.
    • Resident Load usa tabela já carregada.
    • Preceding Load aplica transformação em camadas.
    • QVD é formato nativo e eficiente para recarga.
    • Qlik trabalha fortemente com análise em memória.
    • Expressões usam funções de agregação como Sum(), Count(), Avg().
    • Set Analysis altera o conjunto de dados da agregação.
    • O script é executado na recarga; expressões são avaliadas na análise.
    • Qlik não substitui integralmente SGBD transacional.
    • Chaves artificiais e loops no modelo são pontos clássicos de prova.

    3. Regras Técnicas e Comportamento#

    Script de carga#

    Exemplo básico:

    Vendas:
LOAD
    IdVenda,
    Data,
    Cliente,
    Produto,
    Quantidade,
    ValorUnitario,
    Quantidade * ValorUnitario as ValorTotal
FROM [lib://dados/vendas.xlsx]
(ooxml, embedded labels, table is Plan1);

    LOAD x SELECT#

    • SELECT: consulta na fonte externa.
    • LOAD: transforma, renomeia, calcula, filtra.

    Exemplo combinado:

    SQL SELECT id, nome, cidade
FROM clientes;

LOAD
    id as IdCliente,
    Upper(nome) as NomeCliente,
    cidade as Cidade
;

    Resident Load#

    Usado quando a tabela já está em memória:

    ClientesTratados:
LOAD
    IdCliente,
    NomeCliente,
    Upper(Cidade) as CidadeMaiuscula
Resident Clientes;

    Preceding Load#

    Transformação em camadas, sem precisar de tabela intermediária explícita:

    LOAD
    IdCliente,
    Upper(NomeCliente) as NomeCliente
;
LOAD
    id as IdCliente,
    nome as NomeCliente
FROM [lib://dados/clientes.csv];

    Regras importantes#

    • Campos com mesmo nome em tabelas diferentes serão associados automaticamente.
    • Para evitar associação, renomeie campos.
    • Para forçar concatenação entre tabelas de mesma estrutura, pode-se usar CONCATENATE.
    • Se tabelas tiverem estrutura semelhante, o Qlik pode fazer auto-concatenate.
    • NoConcatenate impede concatenação automática.

    Limitações e cuidados#

    • Modelo mal construído pode gerar:
      • synthetic keys;
      • circular references;
      • ambiguidade analítica.
    • Como é fortemente in-memory, volume de dados e estratégia de carga impactam desempenho.
    • Qlik é excelente para análise, mas não é SGBD OLTP.

    Set Analysis#

    Exemplo:

    Sum({<Ano={2024}, Regiao={'Sul'}>} ValorTotal)

    Significa: somar ValorTotal considerando apenas registros de Ano 2024 e Região Sul, independentemente do estado atual de seleção, conforme a expressão.

    4. Diferenças Críticas#

    Qlik x SQL#

    • SQL: linguagem declarativa para consultar/manipular bancos relacionais.
    • Qlik Script: linguagem de carga e transformação orientada ao motor associativo.
    • Pegadinha: dizer que Qlik substitui SQL completamente está errado.

    Qlik x Power BI/Tableau#

    • Todos são BI/analytics, mas o Qlik se destaca pelo modelo associativo nativo.
    • Em prova, a diferença central é o comportamento de associação automática por nomes de campos.

    Resident Load x Preceding Load#

    • Resident: usa tabela já carregada.
    • Preceding: encadeia loads antes da carga final.
    • Pegadinha: afirmar que são sinônimos é errado.

    Join no Qlik x associação natural do motor#

    • O Qlik pode fazer JOIN, mas o uso excessivo pode prejudicar o modelo.
    • Muitas análises podem ser resolvidas apenas com o modelo associativo, sem join físico.

    5. Exemplos e Aplicações#

    Exemplo de associação#

    Clientes:
LOAD IdCliente, Nome
FROM clientes.csv;

Pedidos:
LOAD IdPedido, IdCliente, Valor
FROM pedidos.csv;

    Como IdCliente tem o mesmo nome nas duas tabelas, o Qlik cria associação automática.

    Exemplo de problema: associação indevida#

    TabelaA:
LOAD Codigo, Descricao
FROM a.csv;

TabelaB:
LOAD Codigo, Valor
FROM b.csv;

    Se Codigo representar entidades diferentes em cada tabela, haverá associação errada. Solução: renomear.

    Exemplo com QVD#

    STORE Vendas INTO vendas.qvd (qvd);

    Uso típico:

    • extrair dados da origem;
    • armazenar em QVD;
    • reutilizar em recargas futuras com maior desempenho.

    Cenários reais#

    • Painéis de vendas com filtros associativos.
    • Análise de inadimplência por cliente, região e período.
    • Consolidação de múltiplas fontes: ERP, planilhas, CSV, banco relacional.

    6. Pegadinhas de Prova#

    • “Qlik associa tabelas por chave primária e estrangeira declaradas no banco.”
      Errado. A associação padrão ocorre por nomes de campos iguais, não por constraints do SGBD.

    • “Synthetic key é recurso obrigatório e desejável.”
      Errado. Pode ocorrer automaticamente, mas em geral indica necessidade de revisão do modelo.

    • “Set Analysis altera os dados da tabela física.”
      Errado. Altera o contexto de cálculo da expressão.

    • “Resident Load consulta diretamente a fonte externa.”
      Errado. Ele usa tabela já carregada em memória.

    • “QVD é banco de dados relacional nativo do Qlik.”
      Errado. É um formato de arquivo otimizado para armazenamento e recarga.

    Termos que invalidam questões#

    Palavras absolutas como:

    • “sempre”
    • “exclusivamente”
    • “substitui totalmente”
    • “necessariamente”
    • “apenas”

    A QUADRIX gosta de inserir absolutizações indevidas.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Modelo associativoRelaciona tabelas por nomes de campos iguaisNão depende de PK/FK declaradas
    LOADComando central de carga e transformaçãoNão é sinônimo de SELECT
    SELECTBusca dados da origem externaSozinho não representa toda a modelagem Qlik
    Resident LoadUsa tabela já carregadaNão lê diretamente da fonte
    Preceding LoadEncadeia transformações em camadasNão é igual a Resident
    QVDArquivo nativo otimizadoNão é SGBD
    Synthetic keySurge com múltiplos campos comunsNão é boa prática desejável
    Set AnalysisDefine conjunto de dados da agregaçãoNão altera tabela física

    8. Simulado (Certo ou Errado)#

    1. No Qlik, a associação entre tabelas ocorre, em regra, automaticamente quando elas possuem campos com o mesmo nome.
    2. O comando Resident Load é utilizado para consultar diretamente uma tabela existente em banco relacional externo.
    3. O formato QVD é otimizado para armazenamento e recarga de dados no ecossistema Qlik.
    4. O uso de Set Analysis modifica permanentemente os registros armazenados no modelo de dados.
    5. Synthetic keys podem surgir quando duas tabelas compartilham múltiplos campos com o mesmo nome.
    6. Em Qlik, LOAD e SELECT são equivalentes e intercambiáveis em qualquer situação.
    7. O modelo associativo do Qlik pode reduzir a necessidade de joins físicos para determinadas análises.
    8. O comando NoConcatenate pode ser usado para impedir concatenação automática de tabelas.

    Gabarito comentado#

    1. Certo.
    Pegadinha: a associação não depende, em regra, de PK/FK declaradas, mas de nomes iguais.

    2. Errado.
    Resident Load opera sobre tabela já carregada em memória.

    3. Certo.
    QVD é formato nativo de alta eficiência para carga/recarga.

    4. Errado.
    Set Analysis só altera o contexto de cálculo da expressão.

    5. Certo.
    Esse é o cenário clássico de surgimento de synthetic keys.

    6. Errado.
    SELECT extrai da origem; LOAD transforma e estrutura. Não são plenamente equivalentes.

    7. Certo.
    O motor associativo permite análises sem exigir todos os joins físicos tradicionais.

    8. Certo.
    NoConcatenate evita que o Qlik una automaticamente tabelas de estrutura semelhante.
