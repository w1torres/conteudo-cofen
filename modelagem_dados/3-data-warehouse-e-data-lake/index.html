    3. Data Warehouse e Data Lake#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Data Warehouse (DW) é um repositório de dados orientado a assunto, integrado, não volátil e variante no tempo, projetado para consulta analítica, apoio à decisão, BI e geração de indicadores. A definição clássica é de Bill Inmon.

    Características centrais do DW:

    • Orientado a assunto: organizado por temas de negócio, como vendas, clientes, finanças.
    • Integrado: consolida dados de múltiplas fontes com padronização.
    • Não volátil: dados são majoritariamente carregados e consultados; alterações transacionais são incomuns.
    • Variante no tempo: mantém histórico.

    Data Lake é um repositório de armazenamento de grande volume que guarda dados em formato bruto ou pouco transformado, estruturados, semiestruturados e não estruturados, para múltiplos usos analíticos, ciência de dados, machine learning e exploração.

    Como funciona internamente#

    Data Warehouse#

    Fluxo típico:

    1. Fontes operacionais: ERP, CRM, bancos OLTP, arquivos, APIs.
    2. ETL/ELT:
      • extração dos dados;
      • limpeza, padronização, deduplicação;
      • transformação para modelo analítico;
      • carga em área intermediária e depois no DW.
    3. Armazenamento analítico:
      • modelagem dimensional (fato e dimensão);
      • agregações, históricos, metadados.
    4. Consumo:
      • OLAP, dashboards, relatórios, consultas SQL analíticas.

    Data Lake#

    Fluxo típico:

    1. Ingestão de dados de diversas fontes.
    2. Armazenamento em arquivos/objetos, muitas vezes com schema flexível.
    3. Processamento sob demanda por engines analíticas.
    4. Consumo por BI, analytics avançado, data science.

    Componentes principais#

    No DW:

    • fontes de dados;
    • staging area;
    • processo ETL/ELT;
    • repositório central;
    • data marts;
    • metadados;
    • camada semântica/BI.

    No Data Lake:

    • camada de ingestão;
    • armazenamento distribuído;
    • catálogo/metadados;
    • motores de processamento;
    • governança e segurança;
    • zonas de dados (raw, trusted, curated).

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Data Warehouse#

    • É voltado a análise, não a processamento transacional.
    • Normalmente suporta consultas complexas e históricas.
    • Usa frequentemente modelagem dimensional.
    • Pode ser alimentado por ETL ou ELT.
    • Pode conter data marts dependentes ou independentes.
    • É base para OLAP.
    • Não é sinônimo de banco relacional comum.
    • Prioriza leitura, consistência analítica e integração.

    Data Lake#

    • Armazena dados brutos ou em baixo nível de transformação.
    • Aceita dados estruturados, semiestruturados e não estruturados.
    • Adota frequentemente schema-on-read.
    • É útil para exploração, ciência de dados e ML.
    • Sem governança, pode virar data swamp.

    Modelagem e armazenamento analítico#

    • Modelo estrela: tabela fato central + dimensões desnormalizadas.
    • Modelo floco de neve: dimensões normalizadas.
    • Tabela fato: medidas e chaves estrangeiras.
    • Tabela dimensão: contexto descritivo.
    • Granularidade define o nível de detalhe da fato.
    • SCD trata histórico em dimensões.

    ETL/ELT#

    • ETL: transforma antes de carregar.
    • ELT: carrega antes e transforma no destino.
    • ETL é tradicional em DW clássico.
    • ELT é comum em plataformas analíticas modernas e lakes/lakehouses.

    3. Regras Técnicas e Comportamento#

    Modelagem: conceitual, lógica e física#

    Conceitual#

    Representa entidades analíticas relevantes:

    • Vendas
    • Cliente
    • Produto
    • Tempo
    • Loja

    Lógica#

    No DW, a modelagem lógica costuma ser dimensional:

    • Fato_Vendas(id_tempo, id_cliente, id_produto, id_loja, quantidade, valor_total)
    • Dim_Cliente(id_cliente, nome, segmento, cidade)
    • Dim_Produto(id_produto, categoria, marca)
    • Dim_Tempo(id_tempo, dia, mês, trimestre, ano)

    Física#

    Inclui:

    • particionamento;
    • índices;
    • compressão;
    • distribuição;
    • colunaridade em bancos analíticos.

    Normalização e desnormalização#

    • Em OLTP, prevalece maior normalização para reduzir redundância.
    • Em DW, é comum desnormalizar dimensões para melhorar desempenho de leitura.
    • Modelo estrela é menos normalizado que floco de neve.

    SQL e comportamento#

    Exemplo analítico em DW:

    SELECT t.ano, p.categoria, SUM(f.valor_total) AS faturamento
FROM fato_vendas f
JOIN dim_tempo t ON f.id_tempo = t.id_tempo
JOIN dim_produto p ON f.id_produto = p.id_produto
GROUP BY t.ano, p.categoria;

    Essa consulta é típica de ambiente analítico: agregação, junções e histórico.

    SCD — Slowly Changing Dimensions#

    Tratamento de mudanças em dimensões:

    • Tipo 1: sobrescreve valor antigo; não preserva histórico.
    • Tipo 2: cria novo registro; preserva histórico.
    • Tipo 3: mantém valor atual e anterior em colunas.

    Limitações e condições de uso#

    DW

    • Menos adequado para dados totalmente não estruturados.
    • Exige modelagem e governança mais rígidas.
    • Pode ter maior custo inicial de projeto.

    Data Lake

    • Sem catálogo e qualidade, dificulta uso corporativo.
    • Não substitui automaticamente um DW para relatórios gerenciais padronizados.
    • Pode ter desempenho ruim sem otimização e organização.

    4. Diferenças Críticas#

    Data Warehouse x Banco OLTP#

    • DW: análise, histórico, leitura intensiva, consultas complexas.
    • OLTP: operação diária, escrita frequente, transações curtas, integridade transacional.

    Data Warehouse x Data Lake#

    • DW:
      • schema-on-write;
      • dados estruturados e tratados;
      • foco em BI e relatórios.
    • Data Lake:
      • schema-on-read;
      • dados brutos e variados;
      • foco mais amplo, inclusive exploração e ML.

    Data Mart x Data Warehouse#

    • Data Mart: subconjunto temático, ex.: financeiro.
    • DW: visão corporativa integrada.

    OLAP x OLTP#

    • OLAP: processamento analítico online.
    • OLTP: processamento transacional online.

    Confusão comum da banca#

    A banca pode sugerir que:

    • Data Lake é sempre superior ao DW: errado.
    • DW armazena apenas dados atuais: errado, ele é histórico.
    • Data Lake exige esquema rígido na carga: errado, em geral prevalece schema-on-read.

    5. Exemplos e Aplicações#

    Exemplo de modelo estrela#

    Tabela fato

    • Fato_Vendas(id_tempo, id_cliente, id_produto, id_loja, qtd, valor)

    Dimensões

    • Dim_Tempo(id_tempo, data, mês, ano)
    • Dim_Cliente(id_cliente, nome, faixa_renda, região)
    • Dim_Produto(id_produto, nome, categoria)
    • Dim_Loja(id_loja, cidade, estado)

    Uso:

    • faturamento por ano;
    • ticket médio por região;
    • vendas por categoria.

    Cenário real#

    Um órgão público integra dados de arrecadação, folha, compras e convênios:

    • no DW, consolida indicadores oficiais e painéis gerenciais;
    • no Data Lake, armazena logs, documentos, arquivos CSV, JSON e dados externos para análises futuras.

    6. Pegadinhas de Prova#

    • “Data Warehouse é voltado a transações operacionais” → errado.
    • “Data Lake armazena apenas dados não estruturados” → errado; armazena também estruturados e semiestruturados.
    • “Modelo estrela é mais normalizado que floco de neve” → errado.
    • “ETL e ELT são sinônimos perfeitos” → errado.
    • “Não volátil” significa que o DW nunca recebe carga” → errado; significa que não sofre atualizações transacionais frequentes como OLTP.
    • “Schema-on-read é típico de DW clássico” → errado; é típico de Data Lake.
    • Termos absolutos como sempre, apenas, exclusivamente, nunca costumam invalidar itens.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Data WarehouseRepositório analítico, integrado e históricoConfundir com banco transacional
    Data LakeArmazena dados brutos de vários tiposDizer que guarda só dados não estruturados
    ETLTransforma antes de carregarTratar como igual a ELT
    ELTCarrega antes e transforma no destinoAfirmar que é incompatível com DW
    Modelo estrelaFato central + dimensões desnormalizadasDizer que é mais normalizado
    Floco de neveDimensões mais normalizadasDizer que é o padrão exclusivo de DW
    SCD Tipo 1Não preserva históricoConfundir com Tipo 2
    SCD Tipo 2Preserva histórico com novo registroDizer que sobrescreve o valor antigo

    8. Simulado (Certo ou Errado)#

    1. O Data Warehouse é tipicamente projetado para suportar consultas analíticas complexas e armazenamento histórico de dados.
    2. No Data Lake, o esquema dos dados deve ser integralmente definido antes da carga, característica conhecida como schema-on-write.
    3. Em modelagem dimensional, a tabela fato normalmente contém medidas de negócio e chaves para dimensões.
    4. O modelo estrela tende a apresentar dimensões mais desnormalizadas que o modelo floco de neve.
    5. O processo ETL caracteriza-se por extrair, carregar e, somente após isso, transformar os dados.
    6. Uma dimensão lentamente mutável do tipo 2 preserva histórico ao inserir novos registros para mudanças relevantes.
    7. Data marts são necessariamente independentes e não podem derivar de um Data Warehouse corporativo.
    8. Um Data Lake mal governado pode degradar-se em um data swamp, dificultando localização e confiabilidade dos dados.

    Gabarito comentado#

    1. Certo.
    Essência do DW: análise, histórico e apoio à decisão.

    2. Errado.
    A pegadinha é confundir Data Lake com DW. Data Lake costuma usar schema-on-read.

    3. Certo.
    Tabela fato = medidas + chaves estrangeiras para dimensões.

    4. Certo.
    Estrela é mais desnormalizado; floco de neve, mais normalizado.

    5. Errado.
    Isso descreve ELT, não ETL. No ETL, transforma-se antes da carga final.

    6. Certo.
    SCD Tipo 2 preserva histórico com novos registros/versionamento.

    7. Errado.
    Data marts podem ser dependentes do DW corporativo.

    8. Certo.
    Sem governança, catálogo e qualidade, o lake vira data swamp.
