    2. Modelagem lógica e física#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Na modelagem de dados, a estrutura de banco de dados é descrita em níveis de abstração:

    • Modelagem conceitual: representa o negócio, com entidades, atributos e relacionamentos, sem preocupação com SGBD.
    • Modelagem lógica: traduz o modelo conceitual para uma estrutura lógica de dados, normalmente relacional, definindo tabelas, colunas, chaves primárias, chaves estrangeiras, domínios e restrições.
    • Modelagem física: implementa o modelo lógico em um SGBD específico, com detalhes de tipos físicos, índices, particionamento, armazenamento, organização de arquivos, constraints e desempenho.

    Como funciona internamente#

    O fluxo clássico é:

    requisitos de negócio → modelo conceitual → modelo lógico → modelo físico → implementação SQL

    Exemplo:

    • Entidade Cliente no conceitual
    • Vira tabela CLIENTE no lógico
    • No físico, recebe tipos concretos: id_cliente BIGINT, nome VARCHAR(100), índice em CPF etc.

    Componentes principais#

    Na estrutura de banco relacional, os principais componentes são:

    • Tabela (relação): conjunto de tuplas/linhas.
    • Coluna (atributo): propriedade da tabela.
    • Linha (tupla/registro): ocorrência de dados.
    • Domínio: conjunto de valores válidos de um atributo.
    • Chave primária (PK): identifica univocamente cada linha.
    • Chave estrangeira (FK): implementa relacionamento e integridade referencial.
    • Chave candidata: atributo(s) que podem identificar univocamente.
    • Chave alternativa: candidata não escolhida como primária.
    • Restrições: NOT NULL, UNIQUE, CHECK, DEFAULT.
    • Índices: aceleram consultas, mas aumentam custo de escrita.
    • Views: tabelas virtuais baseadas em consultas.
    • Sequências/identity: geração de identificadores.
    • Esquema: agrupamento lógico de objetos.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Modelagem conceitual#

    • Entidades fortes e fracas
    • Atributos simples, compostos, multivalorados, derivados
    • Relacionamentos
    • Cardinalidade: 1:1, 1:N, N:N
    • Participação total ou parcial
    • Generalização/especialização

    Modelagem lógica#

    • Conversão de entidades em tabelas
    • Definição de PK
    • Conversão de relacionamentos em FK
    • Resolução de N:N com tabela associativa
    • Normalização
    • Definição de domínios e restrições lógicas

    Modelagem física#

    • Tipos de dados específicos do SGBD
    • Índices
    • Organização de armazenamento
    • Particionamento
    • Clustering
    • Tabelas temporárias/materializadas
    • Ajustes de desempenho

    Normalização#

    • 1FN: eliminar grupos repetitivos e atributos multivalorados
    • 2FN: eliminar dependência parcial da chave
    • 3FN: eliminar dependência transitiva
    • BCNF: todo determinante deve ser chave candidata

    SQL e comportamento#

    • DDL: CREATE, ALTER, DROP
    • DML: INSERT, UPDATE, DELETE
    • DQL: SELECT
    • DCL: GRANT, REVOKE
    • TCL: COMMIT, ROLLBACK

    ETL/ELT#

    • ETL: extrai, transforma, carrega
    • ELT: extrai, carrega, transforma
    • Relação com staging area e DW

    3. Regras Técnicas e Comportamento#

    Regras de transformação para o modelo lógico#

    1. Entidade vira tabela.
    2. Atributo vira coluna.
    3. Identificador da entidade vira PK.
    4. Relacionamento 1:N: FK vai para o lado N.
    5. Relacionamento 1:1: FK em uma das tabelas, preferencialmente no lado de participação total.
    6. Relacionamento N:N: exige tabela associativa com, no mínimo, as FKs das entidades relacionadas.
    7. Atributo multivalorado: não deve permanecer em uma única coluna no modelo relacional normalizado; cria-se nova tabela.
    8. Atributo derivado: em regra, não precisa ser armazenado, salvo necessidade de desempenho/histórico.

    Normalização#

    • 1FN: cada célula deve conter valor atômico; não admite listas em uma coluna.
    • 2FN: só faz sentido quando há chave composta; atributo não-chave não pode depender de parte da chave.
    • 3FN: atributo não-chave não pode depender de outro atributo não-chave.
    • Desnormalização: pode ser adotada por desempenho, mas aumenta redundância e risco de inconsistência.

    Limitações e condições de uso#

    • Modelo lógico é independente do SGBD, mas já depende do paradigma adotado (relacional, por exemplo).
    • Modelo físico é dependente do SGBD.
    • Índice melhora leitura, mas pode degradar INSERT, UPDATE e DELETE.
    • FK garante integridade referencial, mas não impede duplicidade sem UNIQUE.

    4. Diferenças Críticas#

    Conceitual x Lógico x Físico#

    • Conceitual: visão do negócio.
    • Lógico: estrutura formal dos dados.
    • Físico: implementação no SGBD.

    Chave primária x UNIQUE#

    • PK: identifica a linha, é única e não nula.
    • UNIQUE: impede duplicidade, mas em muitos SGBDs pode admitir NULL (dependendo da implementação).

    Relacionamento x chave estrangeira#

    • Relacionamento é conceito de modelagem.
    • FK é mecanismo de implementação no modelo relacional.

    Índice x chave primária#

    • Toda PK costuma gerar índice automaticamente em muitos SGBDs, mas índice não é sinônimo de chave primária.
    • Índice acelera acesso; PK garante unicidade e identificação.

    Normalização x desempenho#

    • Banca confunde ao afirmar que normalização sempre melhora desempenho. Errado.
    • Normalização melhora consistência e reduz redundância; desempenho depende do caso.

    5. Exemplos e Aplicações#

    Exemplo lógico#

    Entidades:

    • Aluno(id_aluno, nome)
    • Curso(id_curso, descricao)

    Relacionamento:

    • Aluno cursa Curso (N:N)

    Implementação lógica/física#

    CREATE TABLE aluno (
    id_aluno INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL
);

CREATE TABLE curso (
    id_curso INT PRIMARY KEY,
    descricao VARCHAR(100) NOT NULL
);

CREATE TABLE aluno_curso (
    id_aluno INT NOT NULL,
    id_curso INT NOT NULL,
    data_matricula DATE NOT NULL,
    PRIMARY KEY (id_aluno, id_curso),
    FOREIGN KEY (id_aluno) REFERENCES aluno(id_aluno),
    FOREIGN KEY (id_curso) REFERENCES curso(id_curso)
);

    Exemplo de 1:N#

    Um departamento possui vários empregados.

    CREATE TABLE departamento (
    id_departamento INT PRIMARY KEY,
    nome VARCHAR(80) NOT NULL
);

CREATE TABLE empregado (
    id_empregado INT PRIMARY KEY,
    nome VARCHAR(80) NOT NULL,
    id_departamento INT NOT NULL,
    FOREIGN KEY (id_departamento) REFERENCES departamento(id_departamento)
);

    ETL/ELT#

    • ETL: transforma antes de carregar no destino.
    • ELT: carrega primeiro no ambiente-alvo e transforma depois, comum em plataformas analíticas modernas.

    6. Pegadinhas de Prova#

    • “Modelo físico é independente do SGBD” → errado.
    • “Relacionamento N:N pode ser implementado diretamente com uma FK” → errado.
    • “Toda coluna UNIQUE pode substituir integralmente uma PK” → errado.
    • “A 2FN elimina dependência transitiva” → errado; isso é 3FN.
    • “Atributo multivalorado atende à 1FN” → errado.
    • “Desnormalização é erro de projeto” → errado; pode ser decisão deliberada.
    • “FK garante que não haverá registros duplicados” → errado.
    • Palavras absolutas como “sempre”, “nunca”, “somente” costumam invalidar assertivas.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Modelo lógicoDefine tabelas, colunas, chaves e relacionamentosNão depende do SGBD específico
    Modelo físicoImplementa no SGBD realDepende do produto e da configuração
    PKÚnica e não nulaNão se confunde com índice
    FKMantém integridade referencialNão garante unicidade
    Relacionamento N:NExige tabela associativaNão vira uma única FK
    1FNValores atômicosLista em coluna viola 1FN
    2FNRemove dependência parcialSó é relevante com chave composta
    3FNRemove dependência transitivaNão é função da 2FN
    ÍndiceAcelera consultaPode piorar escrita
    DesnormalizaçãoPode melhorar desempenhoAumenta redundância

    8. Simulado (Certo ou Errado)#

    1. O modelo lógico de dados descreve a estrutura do banco independentemente de detalhes físicos de armazenamento.
    2. Em um relacionamento muitos-para-muitos, basta inserir uma chave estrangeira em uma das tabelas participantes.
    3. A primeira forma normal exige que os atributos possuam valores atômicos.
    4. A segunda forma normal elimina dependências transitivas entre atributos não-chave.
    5. O modelo físico inclui definição de índices e tipos de dados específicos do SGBD.
    6. Uma chave estrangeira tem como função principal garantir unicidade dos registros da tabela filha.
    7. A desnormalização pode ser adotada por razões de desempenho.
    8. Em relacionamento 1:N, a chave estrangeira normalmente fica na tabela do lado N.

    Gabarito comentado#

    1. Certo.
    Pegadinha: confundir lógico com físico. O lógico abstrai armazenamento.

    2. Errado.
    Pegadinha clássica. N:N exige tabela associativa.

    3. Certo.
    1FN exige atomicidade e ausência de grupos repetitivos.

    4. Errado.
    Pegadinha: dependência transitiva é tratada na 3FN, não na 2FN.

    5. Certo.
    Índices, tipos concretos e armazenamento pertencem ao físico.

    6. Errado.
    FK garante integridade referencial, não unicidade. Para unicidade, usa-se PK ou UNIQUE.

    7. Certo.
    Desnormalização não é necessariamente erro; pode ser estratégia de performance.

    8. Certo.
    No 1:N, a FK vai para o lado muitos.
