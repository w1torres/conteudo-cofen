    4. Bancos relacionais e NoSQL#

    1. Fundamentos e Funcionamento#

    Banco relacional é o SGBD baseado no modelo relacional, com dados organizados em tabelas (relações), linhas (tuplas) e colunas (atributos), manipulados principalmente por SQL. Em prova, relacional se associa a:

    • esquema definido
    • integridade referencial
    • normalização
    • transações ACID
    • joins

    NoSQL é um conjunto de bancos não relacionais ou not only SQL, voltados a flexibilidade de esquema, escalabilidade horizontal e modelos de dados específicos. Principais categorias:

    • chave-valor
    • documento
    • colunar/família de colunas
    • grafo

    Como funciona internamente#

    Nos relacionais, o SGBD gerencia:

    • parser SQL
    • otimizador de consultas
    • executor
    • gerenciador de armazenamento
    • controle de concorrência
    • log de transações
    • buffer cache

    Em geral:

    1. SQL é analisado.
    2. O otimizador escolhe plano de execução.
    3. Índices podem ser usados.
    4. Leituras/escritas passam por mecanismos de bloqueio ou MVCC.
    5. Alterações são registradas em log para recuperação.

    Componentes principais#

    • Tabelas
    • Views
    • Índices
    • Constraints: PK, FK, UNIQUE, CHECK, NOT NULL
    • Stored procedures, functions, triggers
    • Transactions

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Modelagem#

    • Conceitual: entidades, atributos, relacionamentos
    • Lógica: tabelas, chaves, cardinalidades
    • Física: tipos de dados, índices, particionamento, armazenamento

    Normalização#

    • 1FN: eliminar grupos repetitivos; atributos atômicos
    • 2FN: eliminar dependência parcial de chave composta
    • 3FN: eliminar dependência transitiva
    • BCNF: determinante deve ser chave candidata

    SQL#

    • DDL: CREATE, ALTER, DROP
    • DML: INSERT, UPDATE, DELETE
    • DQL: SELECT
    • DCL: GRANT, REVOKE
    • TCL: COMMIT, ROLLBACK, SAVEPOINT

    Integridade#

    • Entidade: PK não nula e única
    • Referencial: FK aponta para PK/UNIQUE válido
    • Domínio: tipo, faixa, formato

    Transações#

    • ACID
      • Atomicidade
      • Consistência
      • Isolamento
      • Durabilidade

    NoSQL#

    • Esquema flexível
    • Escalabilidade horizontal
    • Desnormalização frequente
    • Eventual consistency pode ocorrer
    • Nem todo NoSQL abandona transação; mas geralmente o escopo é menor

    ETL/ELT#

    • ETL: extrai, transforma, carrega
    • ELT: extrai, carrega, transforma no destino
    • Relacionais são comuns em DW; NoSQL pode apoiar data lakes e ingestão massiva

    3. Regras Técnicas e Comportamento#

    Relacionais#

    Oracle, SQL Server, PostgreSQL e MySQL são SGBDs relacionais, mas com diferenças.

    Oracle#

    • Muito cobrado por robustez corporativa.
    • Usa PL/SQL.
    • Forte suporte a transações, particionamento, RAC.
    • Em prova: Oracle é relacional e suporta objetos, mas não deixa de ser relacional.

    SQL Server#

    • Linguagem procedural: T-SQL.
    • Forte integração com ecossistema Microsoft.
    • Recursos clássicos: views, stored procedures, triggers, replicação, Analysis Services.
    • Muito usado em BI tradicional.

    PostgreSQL#

    • Open source, muito aderente a padrões SQL.
    • Suporta MVCC de forma marcante.
    • Tem extensibilidade, tipos avançados, JSON/JSONB.
    • Em prova: possuir JSON não o torna NoSQL; continua relacional.

    MySQL#

    • Muito usado em aplicações web.
    • Open source (com edições comerciais).
    • O comportamento transacional depende do storage engine:
      • InnoDB: transacional, FK, ACID
      • MyISAM: sem suporte pleno a transações e FK
    • Pegadinha clássica: “MySQL sempre suporta integridade referencial” → errado, depende do engine.

    NoSQL: comportamento#

    Chave-valor#

    • Acesso por chave única.
    • Altíssimo desempenho para lookup simples.
    • Ex.: cache, sessão.

    Documento#

    • Armazena JSON/BSON/XML.
    • Esquema flexível.
    • Bom para dados semiestruturados.

    Colunar/família de colunas#

    • Otimizado para grande volume e distribuição.
    • Muito usado em Big Data.

    Grafo#

    • Nós e arestas.
    • Excelente para relacionamentos complexos.

    Limitações#

    Relacional:

    • Escalabilidade horizontal costuma ser mais complexa.
    • Joins pesados podem degradar desempenho.
    • Esquema rígido dificulta mudanças frequentes.

    NoSQL:

    • Pode sacrificar consistência imediata.
    • Menor aderência a joins complexos.
    • Integridade e validação podem migrar para aplicação.

    4. Diferenças Críticas#

    Relacional x NoSQL#

    • Relacional: esquema rígido, SQL, ACID, normalização.
    • NoSQL: esquema flexível, desnormalização, escalabilidade horizontal, modelos variados.

    SQL x SGBD relacional#

    • SQL é linguagem; Oracle, SQL Server, PostgreSQL e MySQL são SGBDs.

    PostgreSQL com JSON x banco NoSQL#

    • PostgreSQL pode armazenar JSON, mas continua sendo relacional multimodelo, não exclusivamente NoSQL.

    MySQL x SQL Server x Oracle x PostgreSQL#

    • Todos são relacionais.
    • Diferem em dialeto, licenciamento, recursos, engine, extensões e administração.
    • A banca pode confundir “open source” com “menos robusto”; isso é falso como regra.

    5. Exemplos e Aplicações#

    Exemplo de modelagem lógica#

    CREATE TABLE cliente (
  id_cliente INT PRIMARY KEY,
  nome VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE
);

CREATE TABLE pedido (
  id_pedido INT PRIMARY KEY,
  id_cliente INT NOT NULL,
  data_pedido DATE NOT NULL,
  valor NUMERIC(10,2),
  CONSTRAINT fk_pedido_cliente
    FOREIGN KEY (id_cliente) REFERENCES cliente(id_cliente)
);

    Exemplo de consulta#

    SELECT c.nome, p.id_pedido, p.valor
FROM cliente c
JOIN pedido p ON c.id_cliente = p.id_cliente
WHERE p.valor > 100;

    Exemplo de desnormalização em NoSQL documental#

    {
  "id_cliente": 10,
  "nome": "Ana",
  "pedidos": [
    {"id_pedido": 1, "valor": 150.00},
    {"id_pedido": 2, "valor": 220.00}
  ]
}

    Aqui, dados de cliente e pedidos podem ficar juntos para leitura rápida, reduzindo joins.

    ETL/ELT#

    • ETL: extrai de Oracle e MySQL, transforma tipos/códigos, carrega em DW SQL Server.
    • ELT: carrega dados brutos em PostgreSQL ou engine analítica e transforma depois.

    6. Pegadinhas de Prova#

    • “NoSQL significa ausência total de SQL.” Errado. É “Not Only SQL”.
    • “Banco relacional exige sempre escalabilidade vertical.” Errado. É comum, mas não exclusivo.
    • “PostgreSQL é NoSQL porque suporta JSON.” Errado.
    • “MySQL sempre é transacional.” Errado; depende do engine.
    • “Normalização melhora sempre o desempenho.” Errado; melhora consistência, mas pode aumentar joins.
    • “NoSQL não possui consistência.” Errado; pode ter, mas muitas soluções priorizam disponibilidade/particionamento.
    • “Oracle, SQL Server, PostgreSQL e MySQL são bancos NoSQL.” Errado.
    • Termos absolutos invalidam questões: sempre, nunca, exclusivamente, apenas.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Banco relacionalUsa tabelas, chaves e SQLSQL é linguagem, não o banco
    NoSQLConjunto de modelos não relacionaisNão significa “sem SQL”
    1FNAtributos atômicosNão trata dependência transitiva
    2FNRemove dependência parcialSó faz sentido com chave composta
    3FNRemove dependência transitivaNão elimina toda redundância possível
    ACIDGarante confiabilidade transacionalNoSQL pode ter transações em certos casos
    PostgreSQLRelacional com recursos avançadosJSON não o torna NoSQL
    MySQLRelacional; engine influencia recursosInnoDB ≠ MyISAM
    FKGarante integridade referencialNem todo engine implementa igualmente
    DesnormalizaçãoPode melhorar leituraPode aumentar redundância e inconsistência

    8. Simulado (Certo ou Errado)#

    1. Os bancos Oracle, SQL Server, PostgreSQL e MySQL são exemplos de SGBDs relacionais.
    2. O suporte a JSON no PostgreSQL o classifica, para fins conceituais, como banco NoSQL e não relacional.
    3. Em bancos relacionais, a integridade referencial é normalmente implementada por meio de chaves estrangeiras.
    4. A 2FN elimina dependências transitivas entre atributos não chave.
    5. NoSQL é uma categoria que pode incluir bancos de chave-valor, documento, colunar e grafo.
    6. No MySQL, o suporte a transações e chaves estrangeiras pode depender do mecanismo de armazenamento utilizado.
    7. A normalização reduz redundância, mas pode aumentar a necessidade de joins em consultas.
    8. Em geral, bancos NoSQL são mais adequados que relacionais para consultas complexas com múltiplos joins e forte integridade referencial.

    Gabarito comentado#

    1. Certo.
    Todos os quatro são SGBDs relacionais clássicos.

    2. Errado.
    Pegadinha: suporte a JSON não altera sua natureza principal. PostgreSQL continua relacional.

    3. Certo.
    FK é o mecanismo típico de integridade referencial.

    4. Errado.
    Pegadinha de normalização: 2FN elimina dependência parcial; dependência transitiva é tema da 3FN.

    5. Certo.
    Essa é a classificação mais cobrada.

    6. Certo.
    No MySQL, isso depende do storage engine; InnoDB suporta, MyISAM não plenamente.

    7. Certo.
    Normalizar melhora consistência, mas pode exigir mais joins.

    8. Errado.
    Em regra, consultas com muitos joins e forte integridade referencial favorecem o modelo relacional, não NoSQL.
