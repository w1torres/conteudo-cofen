# 1. Análise e Projeto de Sistemas

## 1. Definição e Funcionamento

Análise e Projeto de Sistemas é uma disciplina da Engenharia de Software que se concentra em entender os requisitos de negócio e traduzi-los em especificações de software que possam ser implementadas, testadas e mantidas ao longo do tempo. Funciona como uma ponte entre o problema que precisa ser resolvido e a solução tecnológica, envolvendo a modelagem do sistema proposto e a definição de sua arquitetura. É aplicado em qualquer contexto onde um sistema de software precisa ser desenvolvido para atender a necessidades específicas.

## 2. Pontos Críticos de Prova

- **Regras importantes**: Conhecer os diferentes modelos de ciclo de vida de desenvolvimento de software, como cascata, iterativo, ágil, e suas características.
- **Classificações**: Diferenciar entre análise de requisitos, projeto de sistema e projeto de arquitetura.
- **Exceções**: Entender que, embora modelos ágeis enfatizem flexibilidade e adaptação, eles ainda requerem uma forma de análise e projeto, embora menos formalizada.

## 3. Pegadinhas Comuns

- **Confusões frequentes**: Confundir análise de sistemas com design de interface de usuário.
- **Termos que mudam o sentido**: Acreditar que "sempre" deve-se seguir o mesmo modelo de desenvolvimento para todos os projetos.
- **Diferenças entre conceitos próximos**: Não distinguir entre requisitos funcionais e não funcionais.

## 4. Aplicação Prática

Exemplo simples: Em um projeto de software para gerenciamento de biblioteca, a análise de sistemas pode identificar a necessidade de funções como cadastro de usuários, busca de livros e registro de empréstimos. O projeto do sistema então especificaria como essas funções interagem e são estruturadas.

Situação real: Desenvolvimento de um aplicativo de e-commerce, onde a análise de requisitos revela a necessidade de integração com sistemas de pagamento, gestão de inventário e suporte ao cliente.

## 5. Tabela Resumo

| Conceito | Definição | Pegadinha |
|----------|-----------|-----------|
| Análise de Sistemas | Entendimento dos requisitos de negócio | Confundir com design de UI |
| Projeto de Sistema | Tradução de requisitos em especificações técnicas | Achar que é o mesmo que arquitetura de software |
| Requisitos Funcionais | O que o sistema deve fazer | Confundir com requisitos não funcionais |

## 6. Simulado (Certo ou Errado)

1. A análise de sistemas é responsável pela implementação direta das funcionalidades no software. (Errado)
2. Modelos ágeis dispensam completamente a análise e projeto de sistemas. (Errado)
3. Requisitos não funcionais são menos importantes que requisitos funcionais na análise de sistemas. (Errado)
4. O modelo de cascata é incompatível com a revisão de requisitos após o início do desenvolvimento. (Certo)
5. A arquitetura de software é definida durante a fase de análise de requisitos. (Errado)

**Gabarito comentado:**

1. Errado. A análise de sistemas foca na compreensão dos requisitos e na modelagem do sistema, não na implementação.
2. Errado. Modelos ágeis ainda requerem análise e projeto, mas de forma adaptativa e iterativa.
3. Errado. Requisitos não funcionais, como desempenho e segurança, são cruciais para o sucesso do sistema.
4. Certo. O modelo de cascata segue uma abordagem linear e sequencial, dificultando mudanças nos requisitos após o início do projeto.
5. Errado. A definição da arquitetura de software é uma atividade do projeto de sistema, que segue a análise de requisitos.
