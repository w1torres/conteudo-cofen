    4. Teste de Software#

    1. Fundamentos e Funcionamento#

    Definição técnica: teste de software é o processo de verificação e validação de artefatos e do produto executável para identificar defeitos, reduzir riscos e evidenciar conformidade com requisitos. Em prova, atenção: teste não prova ausência de erros, apenas evidencia sua presença e aumenta a confiança.

    No ciclo de vida: ocorre ao longo de todo o desenvolvimento, não apenas no fim. Em abordagens modernas, integra-se ao pipeline de CI/CD, com testes automatizados em múltiplos níveis.

    Relação com V&V:

    • Verificação: “estamos construindo o produto corretamente?”
    • Validação: “estamos construindo o produto correto?”

    Componentes principais do processo de teste:

    • planejamento
    • projeto de casos de teste
    • preparação de massa de dados
    • execução
    • comparação entre resultado esperado e obtido
    • registro de defeitos
    • reteste e regressão

    Classificações centrais cobradas aqui:

    • caixa-preta: baseado em entradas/saídas e requisitos, sem olhar o código
    • caixa-branca: baseado na estrutura interna do código
    • teste de integração: verifica interação entre módulos/componentes
    • teste de validação: verifica aderência às necessidades/requisitos do usuário

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    A. Caixa-preta#

    • Também chamado de teste funcional em muitos contextos
    • Foca no comportamento externo do sistema
    • Não exige conhecimento da implementação interna
    • Técnicas clássicas:
      • particionamento de equivalência
      • análise de valor-limite
      • tabela de decisão
      • grafo de causa e efeito
      • transição de estados
      • casos de uso
    • Pode ser aplicado em vários níveis: unidade, integração, sistema, aceitação

    B. Caixa-branca#

    • Também chamado de teste estrutural
    • Exige conhecimento do código, fluxo de controle e lógica interna
    • Mede cobertura
    • Critérios clássicos:
      • cobertura de instruções/comandos
      • cobertura de decisões/ramificações
      • cobertura de condições
      • cobertura de caminhos (mais forte, geralmente inviável exaustivamente)
    • Muito usado em teste unitário

    C. Integração#

    • Verifica interfaces entre módulos
    • Busca defeitos de comunicação, contrato, formato de dados, temporização
    • Estratégias:
      • big-bang
      • top-down
      • bottom-up
      • sanduíche/híbrida
    • Pode usar stubs e drivers
      • stub simula módulo chamado
      • driver simula módulo chamador

    D. Validação#

    • Confirma se o software atende ao uso pretendido
    • Relaciona-se fortemente a requisitos do usuário
    • Frequentemente associada a teste de sistema e aceitação
    • Pode envolver usuário/cliente
    • Não se confunde com verificação estrutural do código

    3. Regras Técnicas e Comportamento#

    Caixa-preta#

    O testador deriva casos de teste a partir de especificações, histórias de usuário, regras de negócio e contratos de interface.

    Exemplo de valor-limite: campo idade aceita 18 a 60. Casos relevantes:

    • 17, 18, 19
    • 59, 60, 61

    A banca gosta de afirmar que caixa-preta “ignora completamente a estrutura do software”; tecnicamente, não depende dela, mas pode coexistir com conhecimento arquitetural.

    Caixa-branca#

    Analisa o fluxo interno. Exemplo:

    if (idade >= 18 && possuiDocumento) {
    autorizar();
} else {
    negar();
}

    Casos:

    • idade 20, documento true
    • idade 20, documento false
    • idade 17, documento true

    Isso ajuda a cobrir decisões e condições.

    Importante: 100% de cobertura de instruções não implica 100% de cobertura de decisões, nem ausência de defeitos.

    Integração#

    Após testar módulos isoladamente, verifica-se a interação entre eles.

    Falhas típicas:

    • parâmetros em ordem errada
    • tipos incompatíveis
    • API retornando nulo inesperado
    • divergência entre contrato e implementação
    • falhas de serialização, autenticação, timeout

    Estratégias:

    • Big-bang: integra tudo de uma vez. Simples, mas dificulta localizar defeitos.
    • Top-down: começa pelos módulos superiores; usa stubs para inferiores.
    • Bottom-up: começa pelos inferiores; usa drivers para superiores.
    • Híbrida: combina as duas.

    Validação#

    É orientada ao atendimento do propósito do sistema. Ex.: sistema de folha calcula corretamente os descontos legais e gera relatórios exigidos pelo usuário.

    Limitação importante: um software pode passar na verificação técnica e ainda falhar na validação, se não atender à necessidade real do usuário.

    4. Diferenças Críticas#

    Caixa-preta x Caixa-branca#

    • Caixa-preta: testa o que o sistema faz
    • Caixa-branca: testa como o sistema faz

    Integração x Unidade#

    • Unidade: testa componente isolado
    • Integração: testa comunicação entre componentes

    Validação x Verificação#

    • Verificação: conformidade com especificação
    • Validação: adequação ao uso/necessidade

    O que a banca confunde#

    • dizer que teste de caixa-preta é sempre manual → errado
    • dizer que caixa-branca dispensa requisitos → errado
    • dizer que integração ocorre apenas após todos os módulos estarem prontos → errado
    • dizer que validação é sinônimo de homologação → errado; homologação é uma forma/contexto de aceitação, não o conceito inteiro

    5. Exemplos e Aplicações#

    Exemplo arquitetural: sistema em camadas#

    • Controller
    • Service
    • Repository
    • Banco

    Teste de unidade: testa Service isoladamente com mock do Repository.
    Teste de integração: testa Service + Repository + Banco ou integração entre API e banco.
    Teste de caixa-preta: envia requisição HTTP e valida resposta sem olhar o código.
    Teste de caixa-branca: cobre ramos internos do método de cálculo.
    Teste de validação: confirma que o fluxo atende à regra de negócio esperada pelo usuário.

    Exemplo real#

    Em microserviços, teste de integração é crítico para:

    • contratos REST
    • filas/mensageria
    • autenticação entre serviços
    • consistência de payloads

    6. Pegadinhas de Prova#

    • “Teste exaustivo é possível em sistemas pequenos.”
      Geralmente errado; na prática, teste exaustivo é inviável, salvo casos triviais.

    • “Caixa-branca encontra todos os defeitos lógicos.”
      Errado. Aumenta a chance de detectar falhas estruturais, mas não garante completude.

    • “Teste de integração substitui teste unitário.”
      Errado. São complementares.

    • “Validação é feita apenas pelo usuário final.”
      Errado. Pode envolver equipe técnica, cliente, analistas e critérios de aceitação.

    • “Stub e driver são a mesma coisa.”
      Errado.

    • Palavras absolutas que invalidam item: sempre, somente, exclusivamente, garante, elimina.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Caixa-pretaBaseia-se em requisitos e comportamento externoNão é necessariamente manual
    Caixa-brancaBaseia-se na estrutura interna e coberturaCobertura total não garante ausência de defeitos
    IntegraçãoTesta interfaces entre módulosNão ocorre apenas no final
    Big-bangIntegra tudo de uma vezFácil de executar, difícil de diagnosticar
    Top-downComeça de cima para baixoUsa stubs, não drivers
    Bottom-upComeça de baixo para cimaUsa drivers, não stubs
    ValidaçãoVerifica adequação ao uso pretendidoNão é sinônimo estrito de homologação
    VerificaçãoConfirma conformidade com especificaçõesNão responde sozinha se o produto atende ao usuário

    8. Simulado (Certo ou Errado)#

    1. O teste de caixa-preta é elaborado sem necessidade de conhecimento da lógica interna do programa.
    2. O teste de caixa-branca tem como foco principal validar requisitos de negócio sob a ótica do usuário final.
    3. No teste de integração top-down, é comum o uso de stubs para simular módulos inferiores ainda não implementados.
    4. O teste de validação busca demonstrar que o software atende às necessidades para as quais foi desenvolvido.
    5. A obtenção de 100% de cobertura de comandos em um teste de caixa-branca garante que todas as decisões do programa foram exercitadas.
    6. O teste de integração big-bang facilita a localização precisa de defeitos de interface entre módulos.
    7. Drivers são componentes auxiliares usados com frequência em integração bottom-up para simular módulos chamadores.
    8. Testes de caixa-preta podem ser aplicados mesmo quando o código-fonte não está disponível ao testador.

    Gabarito comentado#

    1. Certo.
    Pegadinha: “sem necessidade” não significa “proibido conhecer”; significa que o critério de projeto do teste não depende do código.

    2. Errado.
    Pegadinha: isso descreve mais caixa-preta/validação. Caixa-branca foca estrutura interna.

    3. Certo.
    Top-down desce da camada superior para a inferior; usa stubs para substituir módulos inferiores ausentes.

    4. Certo.
    Essa é a essência da validação: adequação ao uso pretendido.

    5. Errado.
    Cobertura de comandos não implica cobertura de decisões/ramificações.

    6. Errado.
    Big-bang tende a dificultar o isolamento e diagnóstico de falhas.

    7. Certo.
    No bottom-up, como os módulos superiores ainda não existem, usam-se drivers para chamá-los.

    8. Certo.
    Caixa-preta independe da implementação interna; pode ser feito com base em especificações e interface observável.
