    4. Teste de Software#

    1. Fundamentos e Funcionamento#

    O teste de software é uma atividade crítica no desenvolvimento de sistemas, cujo objetivo é identificar falhas e garantir que o software atenda às expectativas de qualidade antes de ser entregue ao usuário final. Envolve a execução do software em condições controladas para verificar se o comportamento observado corresponde ao comportamento esperado.

    Componentes principais:#

    • Casos de Teste: Conjunto de condições ou variáveis sob as quais um testador determinará se um sistema sob teste satisfaz requisitos ou funciona corretamente.
    • Ambiente de Teste: Configuração de hardware e software na qual os testes serão executados, incluindo qualquer equipamento necessário para simular o ambiente de uso do software.
    • Ferramentas de Teste: Softwares que apoiam o processo de teste, como frameworks de automação de testes, ferramentas de gestão de defeitos e simuladores.

    2. Estrutura Completa do Tema (Nível Prova)#

    Classificações de Teste de Software:#

    1. Teste de Caixa Branca (Teste Estrutural): Focado na estrutura interna do software. Requer conhecimento do código fonte.
    2. Teste de Caixa Preta (Teste Funcional): Baseia-se nas especificações do software, sem considerar sua estrutura interna.
    3. Teste de Integração: Verifica a combinação e interação entre diferentes módulos ou serviços.
    4. Teste de Validação: Confirma que o produto final atende às expectativas e requisitos do usuário.

    Exceções:#

    • Teste de Regressão: Não se enquadra diretamente nas categorias anteriores, mas é essencial para garantir que mudanças recentes não afetem funcionalidades existentes.

    3. Regras Técnicas e Comportamento#

    • Teste de Caixa Branca: Utiliza técnicas como cobertura de código, análise de fluxo de controle e de dados. Requer que o testador tenha habilidades de programação.
    • Teste de Caixa Preta: Emprega técnicas como particionamento de equivalência, análise de valor limite e tabelas de decisão.
    • Teste de Integração: Pode ser realizado de maneira incremental (top-down, bottom-up ou sanduíche) ou não incremental.
    • Teste de Validação: Frequentemente inclui testes de aceitação pelo usuário (UAT), onde o foco é verificar se o software atende às necessidades do negócio.

    4. Diferenças Críticas#

    • Caixa Branca vs. Caixa Preta: A principal diferença é o conhecimento do código fonte. Caixa branca exige esse conhecimento, enquanto caixa preta se baseia apenas nas funcionalidades externas.
    • Teste de Integração vs. Validação: Teste de integração foca na combinação de componentes, enquanto a validação verifica a adequação do software para o propósito do usuário final.

    5. Exemplos e Aplicações#

    Exemplo de Teste de Caixa Branca (Código):#

    public int soma(int a, int b) {
    return a + b;
}

    Para testar este método, um teste de caixa branca poderia verificar se a função retorna o resultado correto para valores conhecidos de a e b.

    Cenário de Teste de Caixa Preta:#

    Dado um sistema de cadastro de usuários, um teste de caixa preta poderia incluir a verificação de que o sistema não aceita e-mails sem o caractere “@”.

    6. Pegadinhas de Prova#

    • Erros comuns: Confundir teste de caixa preta com teste de caixa branca.
    • Afirmações que parecem corretas: “Teste de integração não é necessário se os módulos individuais estão funcionando corretamente.” (Falso, pois problemas podem surgir na interação entre módulos.)
    • Termos que invalidam questões: Afirmar que teste de validação é realizado antes do teste de integração invalida uma questão, pois a ordem correta é primeiro integrar e depois validar.

    8. Simulado (Certo ou Errado)#

    1. O teste de caixa branca é realizado sem qualquer conhecimento do código fonte. (Errado)
    2. Teste de caixa preta utiliza técnicas como particionamento de equivalência e análise de valor limite. (Certo)
    3. Teste de integração é desnecessário se os módulos individuais funcionam corretamente. (Errado)
    4. Teste de validação verifica se o produto final atende às expectativas e requisitos do usuário. (Certo)
    5. Teste de regressão se enquadra diretamente como um tipo de teste de caixa preta. (Errado)
    6. Testes de aceitação pelo usuário são uma forma de teste de validação. (Certo)
    7. O teste de caixa preta é mais adequado para encontrar erros de lógica interna do que o teste de caixa branca. (Errado)
    8. Teste de integração pode ser realizado de maneira incremental ou não incremental. (Certo)

    Gabarito comentado:#

    1. Errado. O teste de caixa branca requer conhecimento detalhado do código fonte para sua execução.
    2. Certo. Estas são técnicas comuns utilizadas em testes de caixa preta, focando nas funcionalidades sem considerar a estrutura interna.
    3. Errado. Mesmo que os módulos funcionem isoladamente, o teste de integração é crucial para garantir que eles funcionem corretamente quando combinados.
    4. Certo. O objetivo do teste de validação é assegurar que o software atenda às necessidades e requisitos dos usuários finais.
    5. Errado. O teste de regressão visa identificar novos defeitos em funcionalidades existentes após modificações no software, não se limitando a caixa preta.
    6. Certo. Testes de aceitação pelo usuário são uma parte importante do teste de validação, garantindo que o software atenda às expectativas do usuário.
    7. Errado. O teste de caixa branca é mais eficaz para encontrar erros de lógica interna, pois analisa o código fonte diretamente.
    8. Certo. O teste de integração pode ser adaptado às necessidades do projeto, sendo realizado de forma incremental (adicionando módulos um a um) ou não incremental (todos os módulos de uma vez).
