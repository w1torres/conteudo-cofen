    3. Projeto de Software#

    1. Fundamentos e Funcionamento#

    Projeto de software é a atividade da engenharia de software que transforma requisitos em uma solução técnica estruturada, definindo arquitetura, componentes, interfaces, estruturas de dados e modularidade. É a ponte entre análise e implementação.

    Definição técnica#

    O projeto responde, em essência:

    • como o sistema será organizado;
    • como os dados serão representados e manipulados;
    • como os módulos se relacionam;
    • como atributos de qualidade (manutenibilidade, desempenho, escalabilidade, segurança) serão atendidos.

    Componentes principais#

    1. Arquitetura de software
      Estrutura de alto nível: subsistemas, camadas, estilos arquiteturais, comunicação.
    2. Projeto de dados / estruturas de dados
      Organização lógica dos dados em memória e em trânsito.
    3. Projeto de componentes
      Responsabilidades, interfaces, contratos e dependências.
    4. Projeto de interface
      Interface entre módulos, APIs e, em sentido amplo, interação com usuário.
    5. Modularidade
      Decomposição em partes coesas e fracamente acopladas.

    Como funciona internamente#

    Fluxo típico: requisitos → definição arquitetural → decomposição modular → escolha de estruturas de dados → definição de interfaces → detalhamento de componentes → implementação.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Arquitetura#

    • Define a organização global do sistema.
    • Envolve componentes, conectores e restrições.
    • Pode seguir estilos:
      • Monolítica
      • Em camadas
      • Cliente-servidor
      • MVC
      • SOA
      • Microserviços
      • Event-driven
      • Pipe-and-filter
    • Deve considerar atributos de qualidade:
      • desempenho
      • escalabilidade
      • segurança
      • disponibilidade
      • manutenibilidade
      • testabilidade

    Estruturas de dados#

    • Escolha depende de:
      • tipo de acesso
      • custo de inserção/remoção
      • ordenação
      • consumo de memória
    • Estruturas clássicas:
      • vetor/array
      • lista encadeada
      • pilha (LIFO)
      • fila (FIFO)
      • árvore
      • tabela hash
      • grafo
    • Em projeto, estrutura inadequada compromete desempenho e simplicidade.

    Modularidade#

    • Sistema dividido em módulos com responsabilidade definida.
    • Objetivos:
      • reduzir complexidade
      • facilitar manutenção
      • permitir reuso
      • melhorar testabilidade
    • Conceitos centrais:
      • coesão
      • acoplamento
      • ocultação de informação
      • abstração
      • encapsulamento

    Coesão#

    • Mede o quanto os elementos de um módulo pertencem ao mesmo propósito.
    • Quanto maior, melhor.
    • Tipos clássicos (do pior para o melhor):
      • coincidental
      • lógica
      • temporal
      • procedural
      • comunicacional
      • sequencial
      • funcional

    Acoplamento#

    • Mede dependência entre módulos.
    • Quanto menor, melhor.
    • Tipos clássicos (do pior para o melhor):
      • por conteúdo
      • comum
      • externo
      • por controle
      • por carimbo (stamp)
      • por dados

    3. Regras Técnicas e Comportamento#

    Arquitetura: regras relevantes#

    • Arquitetura não é código detalhado, mas decisão estrutural de alto nível.
    • Monolito: aplicação única, implantação unificada, simples de iniciar, difícil de escalar seletivamente.
    • Microserviços: serviços pequenos e independentes, implantação autônoma, maior complexidade operacional.
    • Camadas: separa apresentação, negócio e dados; favorece organização e manutenção.
    • MVC: separa Model, View e Controller; muito cobrado como padrão arquitetural/padrão de interface.

    Monolito vs microserviços#

    • Monolito:
      • implantação única;
      • transações mais simples;
      • forte acoplamento interno se mal projetado.
    • Microserviços:
      • descentralização;
      • comunicação via rede/API/eventos;
      • tolerância a falhas e observabilidade tornam-se críticas.

    Estruturas de dados: comportamento#

    • Array: acesso indexado rápido; inserções no meio custosas.
    • Lista encadeada: inserção/remoção local facilitada; acesso aleatório ruim.
    • Pilha: útil para chamadas, desfazer, parsing.
    • Fila: útil para processamento assíncrono.
    • Hash: busca eficiente por chave; não preserva ordem naturalmente.
    • Árvore: hierarquia e busca ordenada.
    • Grafo: modela relacionamentos complexos.

    Modularidade: regras centrais#

    • Módulo bom: alta coesão + baixo acoplamento.
    • Encapsulamento: esconder detalhes internos.
    • Ocultação de informação: expor apenas o necessário.
    • Abstração: focar no essencial, omitindo detalhes irrelevantes.
    • Excesso de módulos pode gerar complexidade de integração; modularidade não é fragmentação arbitrária.

    SOLID no projeto#

    • SRP: uma responsabilidade por classe/módulo.
    • OCP: aberto para extensão, fechado para modificação.
    • LSP: subtipos devem substituir tipos base sem quebrar comportamento.
    • ISP: interfaces específicas são melhores que interfaces “gordas”.
    • DIP: depender de abstrações, não de implementações.

    Anti-patterns#

    • God Object/God Class: concentra responsabilidades demais.
    • Spaghetti code: fluxo confuso, baixa modularidade.
    • Shotgun surgery: pequena mudança exige alterações em muitos módulos.
    • Tight coupling: dependência excessiva entre componentes.

    4. Diferenças Críticas#

    Arquitetura x projeto detalhado#

    • Arquitetura: visão macro, decisões estruturais.
    • Projeto detalhado: classes, métodos, algoritmos, estruturas específicas.

    Coesão x acoplamento#

    • Coesão: relação interna do módulo.
    • Acoplamento: relação externa entre módulos.

    Encapsulamento x ocultação de informação#

    • Encapsulamento: agrupar dados e operações.
    • Ocultação de informação: esconder decisões internas de implementação.
      A banca pode tratar como próximos, mas não são idênticos.

    Monolito x sistema modular#

    • Monolito pode ser modular internamente.
    • Microserviços não garantem boa modularidade por si só.

    Estrutura de dados x banco de dados#

    • Estrutura de dados: organização em memória/processamento.
    • Banco de dados: persistência e gerenciamento durável.

    5. Exemplos e Aplicações#

    Exemplo de modularidade ruim#

    class PedidoService {
    void salvarPedido() {}
    void calcularImposto() {}
    void enviarEmail() {}
    void gerarRelatorioFinanceiro() {}
}

    Problema: baixa coesão, responsabilidades misturadas.

    Melhor decomposição#

    class PedidoService {
    void salvarPedido() {}
}

class ImpostoService {
    void calcularImposto() {}
}

class NotificacaoService {
    void enviarEmail() {}
}

    Exemplo arquitetural em camadas#

    • Apresentação: controller REST
    • Negócio: service
    • Persistência: repository/DAO
    • Banco: armazenamento

    Fluxo: Cliente -> Controller -> Service -> Repository -> Banco

    Exemplo de estrutura de dados#

    • Fila para processamento de pedidos:
    Pedido A -> Pedido B -> Pedido C

    Comportamento FIFO: o primeiro a entrar é o primeiro a sair.

    Cenário real#

    Em sistema bancário:

    • arquitetura em camadas para separar regras;
    • fila para processamento assíncrono de notificações;
    • hash para localizar sessão por token;
    • modularidade para isolar pagamento, autenticação e auditoria.

    6. Pegadinhas de Prova#

    • “Maior acoplamento melhora a integração” → errado. Melhora dependência, piora manutenção.
    • “Baixa coesão é desejável para distribuir responsabilidades” → errado.
    • “Microserviços sempre são superiores ao monolito” → errado; depende do contexto.
    • “Encapsulamento elimina a necessidade de interfaces” → errado.
    • “Arquitetura define apenas tecnologia” → errado; define estrutura e relações.
    • “Modularidade garante baixo acoplamento” → errado; pode haver módulos altamente acoplados.
    • “Toda árvore é mais eficiente que tabela hash para busca” → errado.
    • Termos absolutos invalidadores:
      • sempre
      • nunca
      • exclusivamente
      • necessariamente

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    ArquiteturaDefine estrutura de alto nívelNão se limita à linguagem ou framework
    ModularidadeDivide o sistema em partesDividir demais pode aumentar complexidade
    CoesãoQuanto maior, melhorBanca troca com acoplamento
    AcoplamentoQuanto menor, melhorIntegração não exige alto acoplamento
    MonolitoImplantação únicaPode ser bem modularizado
    MicroserviçosServiços independentesTrazem complexidade operacional
    EncapsulamentoOculta detalhes por interfaceNão é sinônimo perfeito de abstração
    FilaFIFOBanca confunde com pilha
    PilhaLIFOBanca inverte com fila
    HashBusca por chave eficienteNão implica ordenação

    8. Simulado (Certo ou Errado)#

    1. A arquitetura de software trata de decisões estruturais de alto nível e da organização dos componentes do sistema.
    2. Em modularidade, o ideal é maximizar o acoplamento e minimizar a coesão para facilitar a comunicação entre módulos.
    3. Um sistema monolítico não pode ser modularizado internamente.
    4. Em geral, pilhas seguem a política LIFO, enquanto filas seguem FIFO.
    5. O princípio da responsabilidade única (SRP) recomenda que um módulo concentre múltiplas responsabilidades correlatas para reduzir a quantidade de classes.
    6. O acoplamento por dados é preferível ao acoplamento por conteúdo.
    7. Microserviços eliminam a necessidade de mecanismos de comunicação entre componentes, pois cada serviço é autônomo.
    8. A escolha de estruturas de dados impacta diretamente desempenho, consumo de memória e complexidade das operações.

    Gabarito comentado#

    1. Certo.
    Pegadinha: arquitetura é visão macro, não apenas implementação.

    2. Errado.
    Correto é alta coesão e baixo acoplamento.

    3. Errado.
    Monolito refere-se sobretudo à implantação/estrutura global; pode haver modularização interna.

    4. Certo.
    Pilha = LIFO; fila = FIFO. Pegadinha clássica.

    5. Errado.
    SRP defende uma única responsabilidade por módulo/classe.

    6. Certo.
    Acoplamento por dados é menos nocivo que por conteúdo, que é dos piores.

    7. Errado.
    Microserviços dependem fortemente de comunicação via rede, APIs, mensageria ou eventos.

    8. Certo.
    Estruturas de dados são decisão central de projeto e afetam comportamento do sistema.
