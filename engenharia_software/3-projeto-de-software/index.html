    3. Projeto de Software#

    1. Fundamentos e Funcionamento#

    O projeto de software é uma fase crítica no desenvolvimento de sistemas, envolvendo a definição da arquitetura, estrutura de dados e modularidade do software. Essa fase determina como o software será estruturado e como os componentes interagem entre si, influenciando diretamente na qualidade, manutenibilidade e expansibilidade do sistema.

    • Definição técnica: O projeto de software é o processo de conceber, especificar, projetar, programar, documentar, testar e corrigir bugs envolvidos na criação e manutenção de aplicações, frameworks ou outros componentes de software.
    • Como funciona internamente: Inicia-se com a coleta de requisitos, seguida pela definição da arquitetura do software, escolha de padrões de projeto e estruturas de dados, e a divisão do sistema em módulos ou componentes menores.
    • Componentes principais: Arquitetura de software, padrões de projeto, estruturas de dados, modularidade.

    2. Estrutura Completa do Tema (Nível Prova)#

    • Arquitetura de Software:
      • Monolítica
      • Orientada a Serviços (SOA)
      • Microserviços
      • Baseada em Eventos
    • Padrões de Projeto (Design Patterns):
      • Padrões de criação (ex: Singleton, Factory)
      • Padrões estruturais (ex: Adapter, Decorator)
      • Padrões comportamentais (ex: Observer, Strategy)
    • Estruturas de Dados:
      • Primitivas (int, float, char)
      • Compostas (arrays, listas, árvores, grafos)
    • Modularidade:
      • Conceito de módulos e componentes
      • Acoplamento e coesão
      • Encapsulamento e ocultação de informação

    3. Regras Técnicas e Comportamento#

    • Arquitetura Monolítica: Toda a aplicação é desenvolvida em um único bloco interdependente. Fácil de desenvolver, mas difícil de escalar.
    • Microserviços: Divide a aplicação em pequenos serviços independentes que comunicam-se através de APIs. Melhora a escalabilidade, mas aumenta a complexidade.
    • Padrões de Projeto: Soluções reutilizáveis para problemas comuns em design de software. Facilitam a manutenção e a expansão do sistema.
    • Estruturas de Dados: Escolha adequada permite otimização de performance e eficiência na manipulação de dados.
    • Modularidade: Módulos bem definidos e com baixo acoplamento aumentam a facilidade de manutenção e a reusabilidade do código.

    4. Diferenças Críticas#

    • Monolito vs Microserviços: Monolitos são mais simples e diretos para projetos pequenos, enquanto microserviços oferecem melhor escalabilidade para sistemas complexos e distribuídos.
    • Padrões GoF vs SOLID: GoF (Gang of Four) são 23 padrões de design focados em soluções para problemas específicos de design, enquanto SOLID são cinco princípios de design orientado a objetos focados em tornar o software mais compreensível, flexível e manutenível.

    5. Exemplos e Aplicações#

    • Padrão Singleton (GoF): Garante que uma classe tenha apenas uma instância e fornece um ponto de acesso global a ela.
      public class Singleton {
    private static Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
    • Princípio SOLID - Single Responsibility Principle: Uma classe deve ter apenas um motivo para mudar, significando que deve ter apenas uma tarefa ou responsabilidade.
      public class Order {
    public void calculateTotalSum() {/*...*/}
    public void getItems() {/*...*/}
    public void getItemCount() {/*...*/}
    public void addItem(Item item) {/*...*/}
    public void deleteItem(Item item) {/*...*/}
}
public class OrderViewer {
    public void printOrder(Order order) {/*...*/}
    public void showOrder(Order order) {/*...*/}
}

    6. Pegadinhas de Prova#

    • Confundir princípios SOLID com padrões GoF.
    • Acreditar que microserviços são sempre a melhor solução.
    • Esquecer que alta coesão é desejável, enquanto alto acoplamento não é.
    • Pensar que padrões de projeto são soluções prontas que funcionam em qualquer contexto.

    8. Simulado (Certo ou Errado)#

    1. Microserviços são recomendados para todos os tipos de projetos, independentemente do seu tamanho e complexidade. (Errado)
    2. O padrão Singleton pode ser aplicado para garantir que uma classe tenha múltiplas instâncias. (Errado)
    3. Alta coesão em módulos significa que eles têm um foco bem definido e limitado. (Certo)
    4. Padrões de projeto são soluções prontas que não necessitam de adaptação ao contexto específico do projeto. (Errado)
    5. O princípio Single Responsibility do SOLID sugere que uma classe deve ter mais de uma responsabilidade. (Errado)
    6. A arquitetura monolítica é mais fácil de escalar do que a baseada em microserviços. (Errado)
    7. Encapsulamento é uma prática que não contribui para a modularidade do software. (Errado)
    8. Padrões comportamentais focam na comunicação eficiente entre objetos. (Certo)

    Gabarito comentado:

    1. Errado. Microserviços são mais adequados para sistemas complexos e que requerem escalabilidade. Para projetos pequenos, a arquitetura monolítica pode ser mais apropriada devido à sua simplicidade.
    2. Errado. O padrão Singleton é utilizado para garantir que uma classe tenha apenas uma instância.
    3. Certo. Alta coesão significa que um módulo ou classe é focado em uma única responsabilidade ou funcionalidade.
    4. Errado. Padrões de projeto fornecem soluções genéricas que devem ser adaptadas ao contexto específico do projeto.
    5. Errado. O princípio Single Responsibility sugere que uma classe deve ter apenas uma responsabilidade.
    6. Errado. A arquitetura monolítica pode ser mais simples de desenvolver inicialmente, mas a baseada em microserviços oferece melhor escalabilidade.
    7. Errado. Encapsulamento é fundamental para a modularidade, pois limita o acesso direto aos componentes internos de um módulo, promovendo a separação de responsabilidades.
    8. Certo. Padrões comportamentais, de fato, focam em como os objetos interagem e distribuem responsabilidades entre si.
