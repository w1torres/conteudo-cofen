    1. Análise e Projeto de Sistemas#

    1. Fundamentos e Funcionamento#

    A análise e projeto de sistemas são etapas fundamentais no desenvolvimento de software, focando na transformação de requisitos em uma arquitetura de sistema operacional. A análise de sistemas envolve a compreensão e especificação do que o sistema deve fazer, enquanto o projeto de sistemas define como o sistema fará para atender aos requisitos especificados.

    • Definição técnica: A análise de sistemas é o processo de entender e especificar em detalhe o que um sistema de software deve fazer. O projeto de sistemas é o processo de especificar a arquitetura, componentes, módulos, interfaces e dados de um sistema para satisfazer os requisitos especificados.
    • Como funciona internamente: Inicia-se com a coleta de requisitos, seguida pela modelagem dos processos de negócio e dados. Após a análise, o projeto é realizado para definir a arquitetura do sistema, escolha de padrões de projeto, definição de módulos e interfaces, e a modelagem física dos dados.
    • Componentes principais: Requisitos, Modelos de Análise (UML, BPMN), Arquitetura de Software (monolítica, microserviços), Padrões de Projeto (GoF, SOLID), e Documentação Técnica.

    2. Estrutura Completa do Tema (Nível Prova)#

    • Regras:
      • A análise de requisitos deve ser completa e clara.
      • O projeto deve seguir padrões de projeto reconhecidos quando aplicável.
    • Classificações:
      • Análise Funcional e Não Funcional
      • Projeto Estrutural e Comportamental
    • Categorias:
      • Padrões de Projeto: Criacionais, Estruturais, Comportamentais
      • Arquitetura de Software: Monolítica, Orientada a Serviços (SOA), Microserviços
    • Exceções:
      • Em alguns casos, a prototipagem rápida pode substituir a análise detalhada.

    3. Regras Técnicas e Comportamento#

    • Funcionamento detalhado: A análise de requisitos deve ser conduzida através de entrevistas, questionários, observação, e análise de documentos existentes. O projeto deve considerar a modularidade, reusabilidade, e manutenibilidade do sistema.
    • Limitações: A análise e projeto de sistemas estão limitados pela precisão dos requisitos coletados e pela capacidade da equipe de interpretá-los corretamente.
    • Condições de uso: Aplicável em projetos de software de qualquer tamanho, mas especialmente crítico em sistemas complexos e de grande escala.

    4. Diferenças Críticas#

    • Comparar com conceitos semelhantes: A análise e projeto de sistemas diferem da programação direta em que se concentram na estruturação e planejamento antes da codificação.
    • Destacar o que a banca confunde: A banca pode confundir análise de sistemas com análise de requisitos, sendo que a análise de sistemas é mais abrangente e inclui a modelagem do sistema.

    5. Exemplos e Aplicações#

    • Exemplos técnicos: Uso de UML para modelar a arquitetura de um sistema de e-commerce, aplicando padrões de projeto como Singleton para a gestão de conexões de banco de dados.
    • Cenários reais: Desenvolvimento de um sistema de gerenciamento de biblioteca, onde a análise de requisitos revela a necessidade de suporte a múltiplas filiais e o projeto define uma arquitetura de microserviços para atender a esta necessidade.

    6. Pegadinhas de Prova#

    • Erros comuns: Confundir análise de sistemas com desenvolvimento de sistemas.
    • Afirmações que parecem corretas: “A análise de sistemas é suficiente para começar a codificação do software.” (Errado, pois o projeto de sistemas também é necessário)
    • Termos que invalidam questões: Uso incorreto de termos técnicos, como referir-se a padrões de projeto como “padrões de codificação”.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Análise de SistemasDeve ser completa e clara, incluindo requisitos funcionais e não funcionais.Confundir com análise de requisitos.
    Projeto de SistemasDeve seguir padrões de projeto reconhecidos e definir claramente a arquitetura.Achar que se limita a escolha de tecnologias.
    Padrões de ProjetoClassificados em Criacionais, Estruturais, Comportamentais.Confundir padrões de projeto com padrões de codificação.

    8. Simulado (Certo ou Errado)#

    1. A análise de sistemas foca exclusivamente nos requisitos funcionais do sistema. (Errado)
    2. O projeto de sistemas pode ser iniciado antes da conclusão da análise de requisitos. (Errado)
    3. Padrões de projeto são aplicáveis apenas no projeto de sistemas de grande escala. (Errado)
    4. A arquitetura de microserviços é uma solução adequada para todos os tipos de sistemas. (Errado)
    5. UML é uma ferramenta exclusiva para a fase de análise de sistemas. (Errado)
    6. A prototipagem rápida pode ser usada como parte da análise de sistemas. (Certo)
    7. Padrões GoF são exemplos de padrões de codificação. (Errado)
    8. A modularidade é um princípio importante tanto na análise quanto no projeto de sistemas. (Certo)

    Gabarito comentado:

    1. Errado. A análise de sistemas inclui tanto requisitos funcionais quanto não funcionais.
    2. Errado. O projeto de sistemas idealmente começa após a análise de requisitos estar completa, para garantir que todos os requisitos sejam adequadamente atendidos.
    3. Errado. Padrões de projeto são úteis em projetos de qualquer escala para resolver problemas comuns de design.
    4. Errado. A arquitetura de microserviços é mais adequada para sistemas complexos e distribuídos, não sendo a solução ideal para todos os casos.
    5. Errado. UML pode ser usada tanto na análise quanto no projeto de sistemas, para modelar diferentes aspectos do sistema.
    6. Certo. A prototipagem rápida é uma técnica útil para validar requisitos e conceitos durante a análise de sistemas.
    7. Errado. Padrões GoF são padrões de projeto, não de codificação, destinados a resolver problemas de design de software.
    8. Certo. A modularidade é fundamental para criar sistemas que são fáceis de entender, modificar, e manter, tanto na análise quanto no projeto.
