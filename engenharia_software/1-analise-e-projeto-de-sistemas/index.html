    1. Análise e Projeto de Sistemas#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Análise de sistemas é a atividade de compreender o problema de negócio, levantar requisitos, modelar o domínio e especificar o que o sistema deve fazer.
    Projeto de sistemas é a atividade de definir como o sistema será construído: arquitetura, componentes, interfaces, dados, comportamento e decisões técnicas.

    Em prova, a distinção central é:

    • Análise → foco no problema e nos requisitos
    • Projeto (design) → foco na solução técnica

    Como funciona internamente#

    O fluxo clássico é:

    1. Levantamento e elicitação de requisitos
    2. Análise
      • identificação de atores, processos, regras de negócio, entidades
      • modelagem funcional, estrutural e comportamental
    3. Projeto
      • definição da arquitetura
      • decomposição em módulos/componentes
      • projeto de dados, interfaces e integrações
    4. Implementação
    5. Testes
    6. Implantação e manutenção

    Componentes principais#

    • Requisitos funcionais: serviços e comportamentos esperados
    • Requisitos não funcionais: desempenho, segurança, escalabilidade, usabilidade
    • Modelos:
      • casos de uso
      • diagramas UML
      • modelos de dados
      • protótipos
    • Arquitetura
    • Padrões de projeto
    • Restrições técnicas e de negócio

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Fundamentos#

    • Análise ≠ projeto
    • Requisito funcional ≠ não funcional
    • Modelo conceitual ≠ modelo lógico ≠ físico
    • Verificação ≠ validação

    Classificações de requisitos#

    • Funcionais: o que o sistema faz
    • Não funcionais: qualidades e restrições
    • De domínio: regras específicas do negócio

    Técnicas de análise#

    • Entrevista
    • Questionário
    • Observação
    • Workshop/JAD
    • Prototipação
    • Análise documental

    Modelagem#

    • UML estrutural: classe, componente, implantação
    • UML comportamental: caso de uso, sequência, atividade, estados
    • DFD e ER ainda podem aparecer como técnicas clássicas

    Projeto#

    • Projeto arquitetural
    • Projeto de dados
    • Projeto de interface
    • Projeto de componentes
    • Projeto de implantação

    Arquitetura#

    • Monolítica
    • Em camadas
    • Cliente-servidor
    • SOA
    • Microserviços
    • Event-driven

    Boas práticas#

    • Alta coesão
    • Baixo acoplamento
    • Separação de responsabilidades
    • Reutilização
    • Testabilidade
    • Manutenibilidade

    Padrões e princípios#

    • SOLID
    • GoF (Factory, Singleton, Observer, Strategy etc.)
    • DRY, KISS, YAGNI

    Anti-patterns#

    • God Object
    • Spaghetti Code
    • Golden Hammer
    • Copy-paste programming
    • Tight coupling

    3. Regras Técnicas e Comportamento#

    Funcionamento detalhado#

    Análise#

    Na análise, o objetivo é abstrair o domínio. O analista identifica:

    • atores externos
    • entradas e saídas
    • processos
    • regras de negócio
    • entidades e relacionamentos
    • eventos do sistema

    A análise não deve antecipar implementação desnecessária. Ex.: dizer “o sistema deve autenticar usuários” é requisito; dizer “usará JWT com RSA-256” já é decisão de projeto.

    Projeto#

    No projeto, define-se a solução técnica:

    • divisão em módulos
    • contratos de interface
    • banco de dados
    • APIs
    • mecanismos de segurança
    • padrões de projeto
    • infraestrutura de implantação

    Arquitetura: comparação essencial#

    • Monolito: aplicação única, implantação única; simples de desenvolver inicialmente, mas pode dificultar escalabilidade e manutenção.
    • Microserviços: serviços independentes, comunicação por API/eventos; maior escalabilidade e autonomia, porém maior complexidade operacional.
    • Camadas: apresentação, negócio, persistência; separa responsabilidades.
    • SOA: serviços corporativos reutilizáveis, frequentemente com barramento/integração.
    • Microserviços não são sinônimo de SOA, embora compartilhem orientação a serviços.

    SOLID#

    • SRP: uma classe deve ter um único motivo para mudar.
    • OCP: aberto para extensão, fechado para modificação.
    • LSP: subtipos devem substituir o tipo base sem quebrar comportamento.
    • ISP: interfaces específicas são melhores que interfaces “gordas”.
    • DIP: depender de abstrações, não de implementações.

    GoF recorrentes em prova#

    • Singleton: uma única instância global.
    • Factory Method: delega criação de objetos.
    • Observer: um-para-muitos; notificação de mudanças.
    • Strategy: encapsula algoritmos intercambiáveis.
    • Facade: interface simplificada para subsistema complexo.
    • Adapter: compatibiliza interfaces incompatíveis.

    Limitações#

    • Excesso de modelagem pode gerar burocracia.
    • Arquitetura complexa sem necessidade viola KISS/YAGNI.
    • Microserviços sem maturidade operacional tendem a aumentar falhas distribuídas.
    • Singleton pode prejudicar testes e aumentar acoplamento global.

    Condições de uso#

    • Use monolito quando simplicidade e entrega rápida forem prioridade.
    • Use microserviços quando houver necessidade real de escalabilidade independente, times autônomos e deploy desacoplado.
    • Use Observer em eventos/notificações.
    • Use Strategy quando houver múltiplas regras intercambiáveis.

    4. Diferenças Críticas#

    Análise x Projeto#

    • Análise: descreve necessidades.
    • Projeto: descreve solução.

    Verificação x Validação#

    • Verificação: “estamos construindo o produto corretamente?”
    • Validação: “estamos construindo o produto certo?”

    Coesão x Acoplamento#

    • Alta coesão é desejável.
    • Baixo acoplamento é desejável. Banca gosta de inverter isso.

    Caso de uso x Cenário#

    • Caso de uso: conjunto de interações para atingir objetivo.
    • Cenário: instância/fluxo específico do caso de uso.

    Classe x Objeto#

    • Classe: molde.
    • Objeto: instância.

    Arquitetura x Padrão de projeto#

    • Arquitetura: estrutura macro do sistema.
    • Padrão de projeto: solução recorrente em nível mais localizado.

    5. Exemplos e Aplicações#

    Exemplo de estrutura em camadas#

    [Apresentação/API]
       |
[Serviço/Negócio]
       |
[Repositório/Persistência]
       |
   [Banco de Dados]

    Exemplo de SRP violado#

    class PedidoService {
    void calcularTotal() {}
    void salvarNoBanco() {}
    void enviarEmail() {}
}

    Problema: uma única classe concentra regras de negócio, persistência e notificação.

    Melhor separação#

    class PedidoService {
    void calcularTotal() {}
}

class PedidoRepository {
    void salvar() {}
}

class EmailService {
    void enviarConfirmacao() {}
}

    Exemplo de Strategy#

    interface Desconto {
    double aplicar(double valor);
}

class DescontoNatal implements Desconto {
    public double aplicar(double valor) { return valor * 0.9; }
}

class DescontoClienteVip implements Desconto {
    public double aplicar(double valor) { return valor * 0.8; }
}

    Permite trocar algoritmo sem alterar o cliente.

    Cenário real#

    Sistema de matrícula:

    • Análise: aluno, curso, turma, pré-requisito, período de matrícula.
    • Projeto: API REST, banco relacional, autenticação, arquitetura em camadas, padrão Repository.

    6. Pegadinhas de Prova#

    • “Requisitos não funcionais descrevem funcionalidades do sistema.” → Errado
    • “Projeto de sistemas define o problema de negócio.” → Errado
    • “Alta coesão e alto acoplamento são desejáveis.” → Errado
    • “Microserviços sempre são melhores que monolitos.” → Errado
    • “UML é linguagem de programação.” → Errado
    • “Caso de uso detalha implementação interna.” → Errado
    • “Singleton melhora necessariamente testabilidade.” → Errado
    • Termos absolutos invalidadores:
      • sempre
      • nunca
      • somente
      • obrigatoriamente

    8. Simulado (Certo ou Errado)#

    1. A análise de sistemas concentra-se na compreensão do domínio e dos requisitos, enquanto o projeto define a estrutura técnica da solução.
    2. Requisitos não funcionais, por não descreverem funções do sistema, não impactam decisões arquiteturais.
    3. Em geral, boa modularização busca alta coesão e baixo acoplamento entre componentes.
    4. O padrão Strategy permite encapsular algoritmos intercambiáveis e variar o comportamento em tempo de execução.
    5. Em arquitetura de microserviços, todos os serviços devem compartilhar o mesmo banco de dados para garantir consistência.
    6. O diagrama de casos de uso da UML representa principalmente interações entre atores e funcionalidades visíveis do sistema.
    7. O princípio Open/Closed afirma que classes devem estar fechadas para extensão e abertas para modificação.
    8. O anti-pattern God Object ocorre quando uma única classe concentra responsabilidades excessivas.

    Gabarito comentado#

    1. Certo.
    Pegadinha: trocar análise por projeto.

    2. Errado.
    Requisitos não funcionais impactam fortemente a arquitetura: desempenho, segurança, disponibilidade etc.

    3. Certo.
    Regra clássica de projeto modular.

    4. Certo.
    Strategy encapsula algoritmos substituíveis.

    5. Errado.
    Microserviços tendem à autonomia; compartilhamento de banco é, em geral, desaconselhado.

    6. Certo.
    Caso de uso mostra visão externa/funcional, não implementação interna.

    7. Errado.
    O correto é: abertas para extensão e fechadas para modificação.

    8. Certo.
    God Object centraliza lógica demais, reduz coesão e aumenta acoplamento.
