    5. Análise de Sistemas#

    1. Definição e Funcionamento#

    A análise de sistemas é o processo de estudar um sistema para identificar seus objetivos e criar sistemas que os atendam, envolvendo a identificação de requisitos, a especificação de sistemas e a modelagem dos processos do sistema. É aplicada no desenvolvimento de sistemas de software para compreender as necessidades do usuário, definir os requisitos do sistema e projetar soluções eficazes.

    2. Pontos Críticos de Prova#

    • Requisitos: São especificações do que um sistema deve fazer e suas restrições. Dividem-se em funcionais (ações do sistema) e não funcionais (desempenho, segurança, etc.).
    • Metodologias: Ágil, Cascata, Espiral, entre outras, cada uma com suas particularidades. A escolha depende do contexto do projeto.
    • Prototipação: Desenvolvimento de versões iniciais de um sistema com o objetivo de testar conceitos ou aspectos específicos.

    3. Pegadinhas Comuns#

    • Confundir requisitos funcionais com não funcionais.
    • Acreditar que a metodologia ágil é aplicável a todos os projetos.
    • Pensar que prototipação é uma fase dispensável.

    4. Aplicação Prática#

    Exemplo de requisito funcional: “O sistema deve permitir o cadastro de usuários.”

    Exemplo de requisito não funcional: “O sistema deve garantir uma resposta em menos de 2 segundos para 95% das transações.”

    5. Tabela Resumo#

    ConceitoDefiniçãoPegadinha

    6. Simulado (Certo ou Errado)#

    1. Requisitos funcionais descrevem como o sistema faz algo, enquanto requisitos não funcionais descrevem o que o sistema faz. (Errado)
    2. A metodologia Ágil é sempre a melhor escolha para todos os projetos de desenvolvimento de software. (Errado)
    3. Prototipação é apenas útil em projetos grandes e complexos. (Errado)
    4. Requisitos não funcionais são menos importantes que requisitos funcionais no desenvolvimento de software. (Errado)
    5. A metodologia Cascata é caracterizada por sua abordagem iterativa e incremental. (Errado)

    Gabarito comentado:#

    1. Errado. Requisitos funcionais descrevem o que o sistema faz (ações), enquanto requisitos não funcionais descrevem como o sistema deve ser (qualidade, desempenho).
    2. Errado. A escolha da metodologia depende do contexto do projeto, incluindo tamanho, complexidade e requisitos específicos.
    3. Errado. Prototipação pode ser útil em qualquer tamanho de projeto para validar conceitos e requisitos com os stakeholders.
    4. Errado. Requisitos não funcionais são tão importantes quanto os funcionais, pois impactam diretamente na usabilidade, segurança, e desempenho do sistema.
    5. Errado. A metodologia Cascata é caracterizada por ser uma abordagem sequencial, onde cada fase deve ser completada antes da próxima começar, não sendo iterativa nem incremental.
