    5. Análise de Sistemas#

    1. Fundamentos e Funcionamento#

    A análise de sistemas é uma disciplina da engenharia de software que se concentra no estudo, compreensão e especificação detalhada dos requisitos de software. Envolve a identificação das necessidades dos stakeholders, a análise de requisitos, a documentação precisa desses requisitos e a comunicação desses requisitos para a equipe de desenvolvimento. Os componentes principais incluem requisitos funcionais, requisitos não funcionais, análise de domínio, modelagem de sistemas e especificação de requisitos.

    2. Estrutura Completa do Tema (Nível Prova)#

    • Requisitos Funcionais e Não Funcionais: Diferenciação entre o que o sistema faz e como ele deve ser.
    • Metodologias de Análise: Inclui abordagens tradicionais (cascata), ágeis (SCRUM, XP) e híbridas.
    • Modelagem de Sistemas: Uso de UML, BPMN para representar visualmente os requisitos e processos.
    • Prototipação: Desenvolvimento de versões iniciais do sistema para validação de requisitos.
    • Validação e Verificação de Requisitos: Processos para assegurar que os requisitos atendem às necessidades e são tecnicamente viáveis.
    • Gestão de Requisitos: Inclui rastreabilidade, priorização e controle de mudanças.

    3. Regras Técnicas e Comportamento#

    • Requisitos Funcionais: Devem ser claros, concisos e testáveis.
    • Requisitos Não Funcionais: Incluem desempenho, segurança, usabilidade, entre outros, e são críticos para a satisfação do usuário.
    • Metodologias Ágeis: Foco na colaboração cliente-desenvolvedor e resposta rápida a mudanças.
    • Prototipação: Pode ser de baixa ou alta fidelidade, dependendo do estágio do projeto e dos recursos disponíveis.

    4. Diferenças Críticas#

    • Análise de Sistemas vs. Desenvolvimento de Sistemas: Análise foca na definição de requisitos e modelagem, enquanto desenvolvimento foca na implementação.
    • Requisitos Funcionais vs. Não Funcionais: Funcionais descrevem ações específicas do sistema, enquanto não funcionais descrevem atributos ou restrições.

    5. Exemplos e Aplicações#

    • Uso de UML para Modelagem: Diagramas de caso de uso para representar requisitos funcionais.
    • Prototipação Rápida: Desenvolvimento de um protótipo de interface de usuário para coletar feedback dos usuários.

    6. Pegadinhas de Prova#

    • Confundir requisitos funcionais com não funcionais: Um erro comum é tratar aspectos de usabilidade (não funcional) como se fossem funcionalidades específicas do sistema.
    • Metodologias Ágeis como solução universal: Nem todos os projetos se beneficiam igualmente das metodologias ágeis; depende do contexto do projeto.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Requisitos FuncionaisDescrevem ações do sistemaConfundidos com características não funcionais
    Requisitos Não FuncionaisDescrevem atributos ou restrições do sistemaVistos como menos importantes
    Metodologias ÁgeisFoco na colaboração e resposta rápida a mudançasConsideradas aplicáveis a qualquer contexto
    PrototipaçãoDesenvolvimento de versões iniciais para validaçãoConsiderada apenas para interfaces de usuário

    8. Simulado (Certo ou Errado)#

    1. ( ) Requisitos não funcionais são menos importantes que os requisitos funcionais na análise de sistemas.
    2. ( ) A prototipação é útil apenas nas fases iniciais do desenvolvimento de software.
    3. ( ) UML é exclusivamente utilizado para a modelagem de requisitos funcionais.
    4. ( ) Metodologias ágeis desencorajam a documentação detalhada de requisitos.
    5. ( ) A validação de requisitos ocorre antes da verificação no ciclo de desenvolvimento de software.
    6. ( ) Requisitos funcionais devem ser testáveis e mensuráveis.
    7. ( ) A análise de sistemas foca exclusivamente na coleta de requisitos dos stakeholders.
    8. ( ) A gestão de requisitos não inclui a rastreabilidade de mudanças ao longo do projeto.

    Gabarito comentado:#

    1. Errado. Requisitos não funcionais são tão importantes quanto os funcionais, pois impactam diretamente a usabilidade, segurança e qualidade do sistema.
    2. Errado. A prototipação pode ser utilizada em qualquer fase do projeto para validar ideias, interfaces ou requisitos.
    3. Errado. UML é uma ferramenta versátil usada para modelar tanto requisitos funcionais quanto não funcionais, além de aspectos estruturais e comportamentais do sistema.
    4. Errado. Metodologias ágeis valorizam a documentação adequada, mas preferem que ela seja o mais eficiente e útil possível, evitando excessos.
    5. Errado. A validação geralmente ocorre após a verificação, assegurando que o produto final esteja alinhado com as necessidades e expectativas dos usuários.
    6. Certo. Requisitos funcionais precisam ser claros, concisos, testáveis e mensuráveis para garantir que possam ser implementados corretamente.
    7. Errado. Além da coleta de requisitos, a análise de sistemas envolve a modelagem, especificação e validação desses requisitos.
    8. Errado. A gestão de requisitos abrange a rastreabilidade, que é essencial para acompanhar as mudanças nos requisitos ao longo do ciclo de vida do projeto.
