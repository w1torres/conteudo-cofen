    3. Scripts de automação#

    1. Fundamentos e Funcionamento#

    Definição técnica:
    Script de automação é um conjunto de instruções executáveis, normalmente interpretadas, destinado a automatizar tarefas repetitivas, operacionais ou de integração entre sistemas. Em concursos, o ponto central é: script automatiza fluxo, reduz intervenção humana, padroniza execução e diminui erro operacional.

    Como funciona internamente:
    Um script recebe entradas, processa lógica e produz saídas/efeitos. Em geral, o fluxo é:

    1. Entrada: parâmetros, arquivos, variáveis de ambiente, APIs, banco de dados.
    2. Processamento: decisões condicionais, laços, chamadas a comandos/sistemas.
    3. Saída: logs, relatórios, alteração de arquivos, envio de mensagens, execução de rotinas.

    Componentes principais:

    • Interpretador: Bash, PowerShell, Python, JavaScript/Node.js etc.
    • Código-fonte: comandos e lógica.
    • Dependências: bibliotecas, módulos, utilitários do SO.
    • Agendador/disparador: cron, Task Scheduler, pipeline CI/CD, webhook.
    • Logs e tratamento de erros: essenciais para rastreabilidade.
    • Credenciais/configuração: variáveis de ambiente, cofres de segredo.

    Características típicas:

    • execução automatizada;
    • repetibilidade;
    • parametrização;
    • integração com SO, rede, arquivos e serviços;
    • possibilidade de execução não interativa.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Classificações dos scripts

    • Por ambiente: local, servidor, nuvem, pipeline.
    • Por linguagem: shell script, PowerShell, Python, Perl, JavaScript.
    • Por finalidade:
      • automação administrativa;
      • provisionamento;
      • backup;
      • monitoramento;
      • ETL simples;
      • integração entre sistemas;
      • testes automatizados;
      • deploy.
    • Por modo de execução:
      • manual;
      • agendado;
      • por evento;
      • sob demanda via API.

    Elementos relevantes

    • parâmetros de entrada;
    • validação de dados;
    • controle de fluxo (if, for, while);
    • tratamento de exceções/erros;
    • logs;
    • idempotência;
    • permissões de execução;
    • dependências;
    • segurança de credenciais;
    • documentação.

    Regras importantes

    • Script pode ser interpretado, não necessariamente compilado.
    • Automação não elimina necessidade de controle e auditoria.
    • Script mal projetado pode propagar erro em escala.
    • Nem todo script é idempotente.
    • Script pode chamar outros programas, APIs e comandos do sistema.
    • Agendamento não é requisito obrigatório para existir automação.

    Exceções e cuidados

    • Scripts interativos são automações limitadas.
    • Scripts com credenciais embutidas são prática insegura.
    • Scripts dependentes de interface gráfica são menos robustos que os baseados em API/CLI.
    • Automação por script não equivale, por si só, a RPA.

    3. Regras Técnicas e Comportamento#

    Funcionamento detalhado#

    Em infraestrutura e operações, scripts costumam atuar como “cola” entre componentes. Exemplo: ler arquivo CSV, validar dados, chamar API REST, gravar log e enviar e-mail.

    Boas práticas#

    • Idempotência: executar várias vezes sem efeito colateral indevido. Ex.: criar diretório apenas se não existir.
    • Modularização: separar funções.
    • Parametrização: evitar valores fixos no código.
    • Tratamento de erro: capturar falhas e retornar códigos adequados.
    • Logging: registrar sucesso, falha, horário e contexto.
    • Princípio do menor privilégio: executar com a menor permissão necessária.
    • Versionamento: manter em Git.
    • Testes: validar em homologação antes de produção.

    Anti-patterns#

    • hardcode de senha;
    • ausência de validação de entrada;
    • dependência de caminho absoluto sem controle;
    • ausência de timeout/retry em chamadas externas;
    • script monolítico e sem comentários;
    • ignorar código de retorno de comandos.

    Limitações#

    • manutenção difícil quando cresce sem arquitetura;
    • dependência do ambiente de execução;
    • fragilidade diante de mudanças em tela/arquivo/formato;
    • baixa governança se não houver versionamento e documentação;
    • risco operacional elevado em scripts destrutivos.

    Condições de uso#

    • ambiente compatível com interpretador;
    • permissões adequadas;
    • dependências instaladas;
    • acesso a recursos externos liberado;
    • variáveis e segredos corretamente configurados.

    4. Diferenças Críticas#

    Script de automação x programa tradicional#

    • Script: geralmente interpretado, rápido para criar, muito usado para tarefas operacionais.
    • Programa tradicional: pode ser compilado, mais estruturado, voltado a aplicações completas.

    Script x RPA#

    • Script: atua preferencialmente por CLI, API, arquivos, banco, SO.
    • RPA: simula ações humanas em interfaces, frequentemente em sistemas sem API.
    • Pegadinha: banca pode sugerir que todo script é RPA. Errado.

    Script x macro#

    • Macro: automação dentro de um aplicativo específico (ex.: Excel VBA).
    • Script: conceito mais amplo, podendo atuar no sistema operacional e em múltiplos serviços.

    Script x workflow/orquestração#

    • Script executa tarefas.
    • Workflow/orquestrador coordena múltiplas etapas, dependências, aprovações e monitoramento.
    • Um workflow pode chamar scripts.

    Shell script x PowerShell#

    • Shell/Bash: comum em Unix/Linux.
    • PowerShell: forte em ambiente Windows e administração orientada a objetos.
    • Pegadinha: PowerShell não manipula apenas texto; trabalha fortemente com objetos.

    5. Exemplos e Aplicações#

    Exemplo em Bash#

    #!/bin/bash
ARQ="/backup/dados.tar.gz"

if [ -f "$ARQ" ]; then
  echo "Backup já existe"
else
  tar -czf "$ARQ" /dados
  if [ $? -eq 0 ]; then
    echo "Backup criado com sucesso"
  else
    echo "Falha no backup" >&2
    exit 1
  fi
fi

    Pontos de prova:

    • #!/bin/bash indica interpretador.
    • if controla fluxo.
    • $? verifica código de retorno do último comando.
    • exit 1 sinaliza erro.

    Exemplo em PowerShell#

    $servico = Get-Service -Name Spooler
if ($servico.Status -ne "Running") {
    Start-Service -Name Spooler
    Write-Output "Serviço iniciado"
} else {
    Write-Output "Serviço já estava em execução"
}

    Ponto crítico: PowerShell manipula objetos retornados por cmdlets.

    Cenários reais#

    • limpeza automática de arquivos temporários;
    • coleta de logs de servidores;
    • criação de usuários em lote;
    • backup e rotação de arquivos;
    • deploy automatizado;
    • reinício controlado de serviços;
    • integração entre planilha, banco e API.

    6. Pegadinhas de Prova#

    • “Todo script é compilado antes da execução.”
      Errado. Em geral, scripts são interpretados.
    • “Automação sempre dispensa supervisão humana.”
      Errado. Pode exigir monitoramento, aprovação ou intervenção.
    • “Script agendado é a única forma de automação.”
      Errado. Pode ser por evento, API, pipeline etc.
    • “Script seguro pode armazenar senha em texto claro, desde que em servidor interno.”
      Errado.
    • “Idempotência significa executar apenas uma vez.”
      Errado. Significa poder repetir sem alterar indevidamente o resultado final.
    • “PowerShell trabalha apenas com saída textual, como shells clássicos.”
      Errado. Trabalha com objetos.
    • “Automação por script elimina a necessidade de logs.”
      Errado. Logs são ainda mais importantes.
    • Termos absolutos invalidadores: “sempre”, “nunca”, “somente”, “exclusivamente”.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    Script de automaçãoAutomatiza tarefas por instruções executáveisNão se limita a tarefas agendadas
    InterpretadorExecuta scripts conforme a linguagemScript não é necessariamente compilado
    IdempotênciaReexecução não gera efeito colateral indevidoNão significa execução única
    LoggingRegistra eventos, erros e auditoriaNão é opcional em ambiente crítico
    PowerShellAdministração com cmdlets e objetosNão manipula apenas texto
    Shell scriptMuito usado em Unix/LinuxNão é exclusivo de Linux
    CredenciaisDevem ficar fora do código, preferencialmente em cofre/variávelSenha hardcoded é falha grave
    RPAAutomação de interface/processo de negócioNão é sinônimo de script

    8. Simulado (Certo ou Errado)#

    1. Scripts de automação são necessariamente programas compilados antes da execução.
    2. Um script pode ser disparado por agendamento, evento ou chamada externa.
    3. Idempotência significa que o script só pode ser executado uma única vez.
    4. Em automação, registrar logs de execução auxilia auditoria e diagnóstico de falhas.
    5. PowerShell diferencia-se de shells tradicionais por trabalhar fortemente com objetos.
    6. Armazenar credenciais diretamente no código-fonte é prática recomendada quando o script roda em rede interna.
    7. Um workflow de automação pode orquestrar vários scripts e etapas dependentes.
    8. Todo script de automação caracteriza, obrigatoriamente, uma solução de RPA.

    Gabarito comentado#

    1. Errado.
    Pegadinha: “necessariamente”. Scripts são, em regra, interpretados.

    2. Certo.
    Pode haver múltiplos gatilhos: cron, evento, webhook, pipeline, execução manual.

    3. Errado.
    Idempotência é reexecução segura, não execução única.

    4. Certo.
    Logs são fundamentais para rastreabilidade, auditoria e troubleshooting.

    5. Certo.
    PowerShell opera com objetos, não apenas com texto bruto.

    6. Errado.
    Rede interna não justifica senha hardcoded. É falha de segurança.

    7. Certo.
    Orquestração coordena scripts, dependências, aprovações e fluxos.

    8. Errado.
    RPA é uma categoria específica, geralmente focada em interface e processo; script é conceito mais amplo.
