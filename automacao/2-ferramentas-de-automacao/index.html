    2. Ferramentas de automação#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Ferramentas de automação de infraestrutura e configuração permitem provisionar, configurar, padronizar e manter ambientes de TI por meio de código, seguindo o paradigma de Infrastructure as Code (IaC) e Configuration Management.

    No contexto de prova, Ansible, Puppet e Chef são ferramentas voltadas principalmente para:

    • automação de configuração;
    • provisionamento;
    • orquestração de tarefas;
    • aplicação de estado desejado em servidores.

    Como funciona internamente#

    Essas ferramentas operam com a ideia de estado desejado: o administrador descreve como o ambiente deve estar, e a ferramenta executa ações para convergir o sistema para esse estado.

    Modelos principais#

    • Declarativo: descreve o estado final desejado.
    • Imperativo/procedural: descreve passos a serem executados.

    Na prática:

    • Ansible: mais associado a automação simples e orquestração; usa playbooks em YAML; execução tipicamente agentless via SSH/WinRM.
    • Puppet: fortemente declarativo; usa manifests em DSL própria; arquitetura clássica com agent/master.
    • Chef: usa DSL baseada em Ruby; historicamente mais procedural, embora suporte convergência declarativa; também usa agent/server.

    Componentes principais#

    Ansible#

    • Control node: máquina que executa o Ansible.
    • Managed nodes: hosts gerenciados.
    • Inventory: lista de hosts/grupos.
    • Playbooks: arquivos YAML com tarefas.
    • Modules: unidades de execução.
    • Roles: organização reutilizável de automação.

    Puppet#

    • Puppet Server/Master: compila catálogos.
    • Agent: roda nos nós.
    • Manifest: definição de recursos.
    • Catalog: estado compilado a aplicar.
    • Facts/Facter: coleta de informações do host.
    • PuppetDB: armazenamento/consulta de dados.

    Chef#

    • Chef Server: centraliza cookbooks, políticas e metadados.
    • Chef Workstation: onde se desenvolve.
    • Node: host gerenciado com chef-client.
    • Cookbooks/Recipes: código de configuração.
    • Resources: abstrações de configuração.
    • Ohai: coleta atributos do sistema.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    • Automação ≠ apenas script; envolve padronização, repetibilidade e controle de estado.
    • Ansible é agentless na forma clássica.
    • Puppet e Chef normalmente usam agente.
    • Ansible usa SSH (Linux/Unix) e WinRM (Windows).
    • Puppet usa manifests e catálogos.
    • Chef usa cookbooks e recipes.
    • Puppet e Chef trabalham com pull model em muitos cenários: agente consulta servidor.
    • Ansible costuma operar em push model: controlador envia comandos aos nós.
    • Todas podem ser usadas em nuvem, on-premises e ambientes híbridos.
    • Idempotência é conceito central: executar várias vezes sem gerar efeitos indevidos.
    • Orquestração = coordenar múltiplas tarefas/sistemas; configuração = ajustar estado de um host/serviço.
    • YAML é típico do Ansible; Puppet e Chef usam DSLs próprias.
    • Chef tem forte associação com Ruby.
    • Puppet é muito cobrado como ferramenta declarativa de gerenciamento de configuração.
    • Ansible é muito cobrado pela simplicidade operacional e ausência de agente.

    3. Regras Técnicas e Comportamento#

    Ansible#

    • Modelo predominante: push.
    • Comunicação: SSH sem agente dedicado no host.
    • Linguagem: YAML.
    • Execução: tarefas em sequência, com módulos.
    • Idempotência: depende do módulo; módulos bem projetados evitam alterações desnecessárias.
    • Vantagens:
      • baixa barreira de entrada;
      • sem necessidade de instalar agente;
      • bom para orquestração e tarefas ad hoc.
    • Limitações:
      • em escala muito alta, pode haver gargalo no nó controlador;
      • depende de conectividade remota e credenciais adequadas;
      • nem toda automação é naturalmente declarativa.

    Exemplo:

    - hosts: web
  become: yes
  tasks:
    - name: Instalar nginx
      apt:
        name: nginx
        state: present

    Puppet#

    • Modelo predominante: pull, com agente consultando o servidor.
    • Linguagem: DSL declarativa.
    • Fluxo:
      1. agent coleta facts;
      2. envia ao server;
      3. server compila catálogo;
      4. agent aplica catálogo.
    • Forte foco em estado desejado e conformidade contínua.
    • Vantagens:
      • robusto para grandes ambientes;
      • boa governança de configuração;
      • convergência periódica.
    • Limitações:
      • curva de aprendizado maior;
      • dependência de infraestrutura de servidor/agente.

    Exemplo:

    package { 'nginx':
  ensure => installed,
}

service { 'nginx':
  ensure => running,
  enable => true,
}

    Chef#

    • Modelo comum: pull, com chef-client no nó.
    • Linguagem: DSL em Ruby.
    • Estrutura:
      • cookbook → conjunto de receitas;
      • recipe → sequência lógica de recursos;
      • resource → unidade declarativa/operacional.
    • Vantagens:
      • alta flexibilidade;
      • forte capacidade de customização;
      • integração com ecossistema DevOps.
    • Limitações:
      • maior complexidade;
      • exige familiaridade com Ruby/DSL.

    Exemplo:

    package 'nginx' do
  action :install
end

service 'nginx' do
  action [:enable, :start]
end

    4. Diferenças Críticas#

    Ansible x Puppet x Chef#

    • Ansible: sem agente, push, YAML, simplicidade.
    • Puppet: com agente, declarativo, catálogo, conformidade contínua.
    • Chef: com agente, DSL Ruby, flexível e programável.

    Configuração x Orquestração#

    • Configuração: instalar pacote, editar arquivo, iniciar serviço.
    • Orquestração: coordenar vários sistemas em ordem lógica, por exemplo:
      1. criar VM;
      2. configurar banco;
      3. subir aplicação;
      4. atualizar balanceador.

    IaC x Script tradicional#

    • Script tradicional: sequência de comandos, muitas vezes sem controle de estado.
    • IaC: descrição versionável, reexecutável, auditável e preferencialmente idempotente.

    O que a banca confunde#

    • Dizer que Ansible exige agente → errado.
    • Dizer que Puppet é tipicamente agentless → errado.
    • Dizer que Chef usa YAML como linguagem principal → errado.
    • Dizer que idempotência significa executar apenas uma vez → errado.

    5. Exemplos e Aplicações#

    Cenários reais#

    • Padronizar configuração de 500 servidores Linux.
    • Instalar e configurar Apache/Nginx em lote.
    • Criar usuários, chaves SSH e políticas de acesso.
    • Aplicar hardening básico em servidores.
    • Provisionar infraestrutura em nuvem e depois configurar serviços.

    Fluxo típico com Ansible#

    1. Definir inventário.
    2. Criar playbook.
    3. Executar do nó controlador.
    4. Hosts recebem comandos via SSH.
    5. Resultado consolidado no controlador.

    Fluxo típico com Puppet/Chef#

    1. Nó possui agente.
    2. Agente coleta dados locais.
    3. Consulta servidor central.
    4. Recebe política/catálogo/receita.
    5. Aplica e reporta estado.

    6. Pegadinhas de Prova#

    • “Ansible é exclusivamente declarativo”: errado. Ele suporta abordagem declarativa em muitos módulos, mas também executa tarefas procedurais.
    • “Puppet e Chef operam apenas em modo push”: errado. O padrão clássico é pull via agente.
    • “Agentless significa sem qualquer software no host”: cuidado. Não exige agente dedicado, mas exige mecanismos como SSH/Python/PowerShell conforme o caso.
    • “Idempotência impede reexecução”: errado. Permite reexecução sem efeitos colaterais indevidos.
    • “Cookbooks pertencem ao Puppet”: errado; são do Chef.
    • “Manifest é termo típico do Ansible”: errado; é do Puppet.
    • “Playbook é termo do Chef”: errado; é do Ansible.

    Termos absolutos que invalidam itens:

    • “sempre”
    • “exclusivamente”
    • “necessariamente”
    • “apenas”
    • “obrigatoriamente”

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    AnsibleNormalmente agentless e pushNão confundir com Puppet/Chef
    PuppetUsa agente, manifests e catálogosNão usa playbooks
    ChefUsa cookbooks/recipes e DSL RubyNão usa YAML como principal
    IdempotênciaReexecução sem alteração indevidaNão significa execução única
    OrquestraçãoCoordena múltiplas etapas/sistemasNão é sinônimo estrito de configuração
    InventoryLista hosts/grupos no AnsibleNão é catálogo do Puppet
    Facts/FacterMetadados do host no PuppetNão confundir com Ohai
    OhaiColeta atributos no ChefNão pertence ao Puppet

    8. Simulado (Certo ou Errado)#

    1. O Ansible, em sua forma clássica, gerencia hosts remotos sem exigir a instalação de agente dedicado nesses hosts.
    2. O Puppet utiliza playbooks em YAML para compilar catálogos aplicados pelos agentes.
    3. No Chef, cookbooks podem conter recipes que definem recursos e ações de configuração.
    4. O modelo de operação mais comum do Ansible é pull, no qual os nós consultam periodicamente o servidor central.
    5. Idempotência, em automação, significa que a mesma configuração pode ser reaplicada sem produzir mudanças indevidas quando o estado desejado já foi atingido.
    6. Puppet e Chef são exemplos clássicos de ferramentas que frequentemente utilizam agentes instalados nos nós gerenciados.
    7. No Ansible, o inventário serve para definir hosts e agrupamentos sobre os quais playbooks podem atuar.
    8. Chef é uma ferramenta cuja DSL está associada à linguagem Ruby, diferentemente do Ansible, que usa majoritariamente YAML.

    Gabarito comentado#

    1. Certo.
    Pegadinha: “agentless” é característica clássica do Ansible.

    2. Errado.
    Pegadinha: Puppet usa manifests em DSL própria, não playbooks em YAML.

    3. Certo.
    Cookbooks e recipes são elementos centrais do Chef.

    4. Errado.
    Pegadinha: Ansible é tipicamente push, não pull.

    5. Certo.
    Essa é a definição prática de idempotência em automação.

    6. Certo.
    Ambos, em arquiteturas clássicas, usam agentes nos nós.

    7. Certo.
    Inventory no Ansible define hosts, grupos e variáveis associadas.

    8. Certo.
    Chef usa DSL baseada em Ruby; Ansible usa YAML em playbooks.
