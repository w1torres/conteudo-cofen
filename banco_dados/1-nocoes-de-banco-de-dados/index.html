    1. Noções de banco de dados#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    Banco de dados (BD) é uma coleção organizada de dados relacionados, armazenados de forma estruturada, com o objetivo de permitir armazenamento, recuperação, atualização e gerenciamento eficiente das informações.

    SGBD (Sistema de Gerenciamento de Banco de Dados) é o software que permite definir, manipular, controlar e administrar o banco de dados. Exemplos: PostgreSQL, Oracle, MySQL, SQL Server.

    Atenção de prova:

    • Banco de dados = conjunto de dados.
    • SGBD = software que gerencia esses dados.
      A banca costuma trocar os conceitos.

    Como funciona internamente#

    Em termos gerais, o funcionamento envolve:

    1. Definição da estrutura dos dados (esquema).
    2. Armazenamento físico em arquivos, páginas, blocos ou segmentos.
    3. Manipulação por comandos, principalmente SQL.
    4. Controle de concorrência para múltiplos usuários.
    5. Controle de integridade para manter consistência.
    6. Recuperação em caso de falhas, com logs e backup.

    Componentes principais#

    • Dados: fatos armazenados.
    • Metadados: dados sobre os dados (estrutura, tipos, restrições).
    • Esquema: descrição lógica do banco.
    • Instância: conteúdo do banco em um momento específico.
    • Tabelas/relacionamentos: estrutura típica do modelo relacional.
    • Índices: aceleram consultas.
    • Visões (views): tabelas virtuais.
    • Transações: unidades lógicas de trabalho.
    • Catálogo/dicionário de dados: repositório de metadados.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de conceitos essenciais#

    • Dado: valor bruto, sem interpretação isolada.
    • Informação: dado processado com significado.
    • Banco de dados: coleção organizada de dados relacionados.
    • SGBD: software de gerenciamento do BD.
    • Sistema de banco de dados: BD + SGBD + aplicações + usuários.
    • Esquema: estrutura definida do banco.
    • Instância: estado atual dos dados.
    • Modelo de dados: forma de representar dados e relacionamentos.
    • Abstração de dados: ocultação de detalhes de implementação.
    • Independência de dados: alteração em um nível sem afetar outro.

    Classificações importantes#

    Quanto ao modelo de dados#

    • Hierárquico: estrutura em árvore; relação pai-filho.
    • Em rede: registros com múltiplas ligações.
    • Relacional: dados em tabelas; modelo mais cobrado.
    • Orientado a objetos: integra conceitos de OO.
    • NoSQL: não relacional; chave-valor, documento, coluna, grafo.

    Quanto à arquitetura#

    • Centralizado: banco em um único local.
    • Distribuído: dados distribuídos em vários nós.
    • Cliente-servidor: clientes acessam servidor de BD.

    Modelagem de dados#

    • Conceitual: visão de negócio; entidades, atributos, relacionamentos.
    • Lógica: tradução para estrutura lógica, sem foco físico.
    • Física: implementação no SGBD, tipos, índices, armazenamento.

    Normalização#

    Processo para reduzir redundância e anomalias.

    • 1FN: atributos atômicos; sem grupos repetitivos.
    • 2FN: estar na 1FN e eliminar dependência parcial de chave composta.
    • 3FN: estar na 2FN e eliminar dependência transitiva.
    • BCNF: versão mais forte da 3FN; todo determinante deve ser chave candidata.

    Linguagens SQL#

    • DDL: definição (CREATE, ALTER, DROP)
    • DML: manipulação (INSERT, UPDATE, DELETE)
    • DQL: consulta (SELECT)
    • DCL: controle (GRANT, REVOKE)
    • TCL: transações (COMMIT, ROLLBACK, SAVEPOINT)

    ETL/ELT#

    • ETL: Extract, Transform, Load.
    • ELT: Extract, Load, Transform. Diferença central: no ETL a transformação ocorre antes da carga; no ELT, depois, geralmente no destino.

    3. Regras Técnicas e Comportamento#

    Níveis de abstração#

    Modelo clássico ANSI/SPARC:

    • Nível externo: visão do usuário.
    • Nível conceitual: visão global lógica.
    • Nível interno: armazenamento físico.

    Independência de dados#

    • Lógica: mudanças no nível conceitual não afetam visões externas.
    • Física: mudanças no armazenamento físico não afetam o nível lógico.

    Integridade#

    • Integridade de entidade: chave primária não pode ser nula.
    • Integridade referencial: chave estrangeira deve referenciar chave válida ou ser nula, se permitido.
    • Integridade de domínio: valores devem respeitar tipo, formato e restrições.

    Transações e ACID#

    • Atomicidade: tudo ou nada.
    • Consistência: leva o BD de um estado válido a outro.
    • Isolamento: transações concorrentes não devem interferir indevidamente.
    • Durabilidade: após confirmação, os dados persistem.

    Limitações e condições#

    • Banco relacional é excelente para consistência e consultas estruturadas, mas pode ter limitações de escalabilidade horizontal em certos cenários.
    • Bancos NoSQL priorizam, em muitos casos, flexibilidade e escalabilidade, mas podem abrir mão de parte da consistência forte, dependendo do modelo.

    4. Diferenças Críticas#

    Banco de dados x SGBD#

    • BD = dados armazenados.
    • SGBD = software que gerencia.

    Esquema x Instância#

    • Esquema = estrutura.
    • Instância = conteúdo atual.

    Dado x Informação#

    • Dado = elemento bruto.
    • Informação = dado interpretado.

    Chave primária x chave estrangeira#

    • Primária: identifica unicamente um registro.
    • Estrangeira: referencia chave de outra tabela.

    Modelagem conceitual x lógica x física#

    • Conceitual: negócio.
    • Lógica: estrutura do modelo.
    • Física: implementação real.

    SQL x Banco relacional#

    SQL é a linguagem; banco relacional é o modelo/estrutura.
    Nem todo banco usa SQL da mesma forma, e SQL não se confunde com o próprio banco.

    5. Exemplos e Aplicações#

    Exemplo de modelagem simples#

    Entidades:

    • Cliente(id_cliente, nome, cpf)
    • Pedido(id_pedido, data, id_cliente)

    Relacionamento:

    • Um cliente pode fazer vários pedidos.
    • Um pedido pertence a um cliente.

    Exemplo SQL#

    CREATE TABLE Cliente (
    id_cliente INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    cpf CHAR(11) UNIQUE
);

CREATE TABLE Pedido (
    id_pedido INT PRIMARY KEY,
    data_pedido DATE NOT NULL,
    id_cliente INT,
    FOREIGN KEY (id_cliente) REFERENCES Cliente(id_cliente)
);

    Exemplo de consulta#

    SELECT c.nome, p.id_pedido
FROM Cliente c
JOIN Pedido p ON c.id_cliente = p.id_cliente;

    Exemplo de ETL#

    1. Extrair dados de planilhas.
    2. Padronizar CPF e datas.
    3. Carregar em tabela de clientes.

    Exemplo de problema sem normalização#

    Tabela única com cliente e pedido repetindo nome e CPF em cada linha gera:

    • Redundância
    • Anomalia de atualização
    • Anomalia de inserção
    • Anomalia de exclusão

    6. Pegadinhas de Prova#

    • “Banco de dados é o software que gerencia os dados.”
      Errado. Isso é o SGBD.

    • “Esquema é o conteúdo atual do banco.”
      Errado. Isso é a instância.

    • “Chave estrangeira identifica unicamente cada registro da tabela.”
      Errado. Essa função é da chave primária.

    • “Normalização sempre melhora desempenho.”
      Errado. Normalização melhora consistência e reduz redundância; desempenho pode exigir desnormalização.

    • “SQL é apenas linguagem de consulta.”
      Errado. SQL inclui definição, manipulação, controle e transação.

    • Palavras perigosas: “sempre”, “somente”, “exclusivamente”, “nunca”.
      Em banco de dados, costumam invalidar assertivas.

    Chave estrangeiraReferencia chave de outra tabelaDizer que identifica unicamente

    8. Simulado (Certo ou Errado)#

    1. O banco de dados corresponde ao software responsável por controlar acesso, concorrência e integridade dos dados.
    2. No modelo relacional, os dados são organizados, em regra, em tabelas compostas por linhas e colunas.
    3. Esquema é a descrição da estrutura do banco, enquanto instância é o conjunto de dados armazenados em determinado momento.
    4. A integridade de entidade estabelece que uma chave estrangeira deve sempre referenciar uma chave primária válida.
    5. A 1FN exige que os atributos sejam atômicos e que não existam grupos repetitivos.
    6. A linguagem SQL abrange apenas comandos de consulta, como o SELECT.
    7. Em ETL, a transformação dos dados ocorre antes da carga no destino.
    8. A independência física de dados permite alterar a forma de armazenamento sem necessidade de alterar o esquema lógico.

    Gabarito comentado#

    1. Errado.
    Pegadinha: a descrição é do SGBD, não do banco de dados.

    2. Certo.
    É a característica central do modelo relacional.

    3. Certo.
    Definição clássica e muito cobrada.

    4. Errado.
    A assertiva descreve integridade referencial, não integridade de entidade.

    5. Certo.
    Regra formal da Primeira Forma Normal.

    6. Errado.
    SQL não é só consulta; inclui DDL, DML, DCL e TCL.

    7. Certo.
    No ETL, transforma-se antes de carregar.

    8. Certo.
    Essa é exatamente a ideia de independência física.
