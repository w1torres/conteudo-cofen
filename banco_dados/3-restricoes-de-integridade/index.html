    3. Restrições de integridade#

    1. Fundamentos e Funcionamento#

    Definição técnica:
    Restrições de integridade são regras impostas ao banco de dados para garantir correção, consistência e validade dos dados armazenados. Elas impedem que o estado do banco viole regras do modelo relacional ou regras de negócio implementadas no esquema.

    No modelo relacional, integridade significa que os dados devem permanecer coerentes entre si, mesmo após inserções, atualizações e exclusões.

    Como funciona internamente#

    As restrições são definidas no esquema da tabela e verificadas pelo SGBD durante operações de:

    • INSERT
    • UPDATE
    • em certos casos, DELETE (especialmente integridade referencial)

    Se a operação violar uma restrição, o SGBD:

    • rejeita a operação; ou
    • aplica ação configurada, como CASCADE, SET NULL, SET DEFAULT.

    Componentes principais#

    • Chave primária (PRIMARY KEY)
    • Chave candidata
    • Chave alternativa
    • Chave estrangeira (FOREIGN KEY)
    • UNIQUE
    • NOT NULL
    • CHECK
    • DEFAULT (não é, estritamente, uma restrição de integridade lógica forte, mas ajuda na consistência)
    • Integridade de domínio
    • Integridade de entidade
    • Integridade referencial

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    A. Classificações das restrições#

    • Integridade de domínio: valores devem pertencer a um domínio válido (tipo, faixa, formato, conjunto permitido).
    • Integridade de entidade: toda tupla deve ser identificável unicamente; chave primária não pode ser nula.
    • Integridade referencial: chave estrangeira deve referenciar chave existente na tabela pai, ou ser nula se permitido.
    • Integridade semântica: regras específicas do negócio, muitas vezes implementadas com CHECK, triggers ou lógica da aplicação.

    B. Chaves#

    • Superchave: conjunto de atributos que identifica unicamente uma tupla.
    • Chave candidata: superchave mínima.
    • Chave primária: uma das chaves candidatas escolhida como principal.
    • Chave alternativa: chave candidata não escolhida como primária.
    • Chave composta: formada por mais de um atributo.
    • Chave estrangeira: atributo(s) que referencia(m) chave de outra relação.

    C. Regras formais#

    • Chave primária: única e não nula.
    • UNIQUE: garante unicidade, mas em muitos SGBDs pode admitir NULL conforme implementação.
    • NOT NULL: impede valor nulo.
    • CHECK: impõe condição booleana.
    • FOREIGN KEY: exige correspondência com valor existente na tabela referenciada.

    D. Exceções e observações#

    • Uma tabela possui apenas uma chave primária.
    • Uma tabela pode ter várias chaves candidatas.
    • Uma tabela pode ter várias restrições UNIQUE.
    • Chave estrangeira não precisa ser única.
    • Chave estrangeira pode ser composta.
    • Chave estrangeira pode referenciar, em geral, PRIMARY KEY ou UNIQUE.

    3. Regras Técnicas e Comportamento#

    Modelagem e integridade#

    Conceitual#

    No modelo ER, a integridade aparece na identificação de:

    • entidades
    • atributos identificadores
    • relacionamentos obrigatórios/opcionais
    • cardinalidades

    Lógica#

    No modelo relacional:

    • entidades viram tabelas
    • identificadores viram chaves primárias
    • relacionamentos viram chaves estrangeiras

    Física#

    No SGBD, as restrições são materializadas com DDL.

    Exemplo SQL#

    CREATE TABLE Departamento (
    id_departamento INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL UNIQUE
);

CREATE TABLE Empregado (
    id_empregado INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    salario DECIMAL(10,2) CHECK (salario >= 0),
    id_departamento INT,
    CONSTRAINT fk_emp_dep
        FOREIGN KEY (id_departamento)
        REFERENCES Departamento(id_departamento)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);

    Comportamento das principais restrições#

    PRIMARY KEY#

    • identifica univocamente cada linha;
    • implica UNIQUE + NOT NULL;
    • pode ser simples ou composta.
    PRIMARY KEY (id)

    UNIQUE#

    • impede duplicidade;
    • difere da chave primária porque não é, necessariamente, a identificação principal;
    • tratamento de NULL varia por SGBD, mas em provas costuma-se cobrar que não equivale à chave primária.

    FOREIGN KEY#

    • mantém consistência entre tabelas;
    • evita registros “órfãos”;
    • ações comuns:
      • RESTRICT / NO ACTION
      • CASCADE
      • SET NULL
      • SET DEFAULT

    CHECK#

    • valida condição local à linha;
    • exemplo:
    CHECK (idade >= 18)
    • em geral, não cruza tabelas no padrão relacional clássico de forma robusta como uma FK.

    NOT NULL#

    • impede ausência de valor;
    • importante: NULL não é zero, nem string vazia, nem falso.

    Normalização e integridade#

    A normalização reduz anomalias e reforça consistência:

    • 1FN: atributos atômicos, sem grupos repetitivos.
    • 2FN: elimina dependência parcial de chave composta.
    • 3FN: elimina dependência transitiva entre atributos não chave.
    • BCNF: todo determinante deve ser chave candidata.

    A banca pode associar incorretamente normalização com “substituição” de restrições. Errado: normalização ajuda, mas não elimina a necessidade de PK, FK, CHECK etc.

    ETL/ELT#

    Em processos de carga:

    • ETL pode validar dados antes da carga;
    • ELT pode carregar e validar no destino;
    • restrições podem ser temporariamente desabilitadas em cargas massivas, mas isso não elimina sua importância.

    4. Diferenças Críticas#

    PRIMARY KEY x UNIQUE#

    • PRIMARY KEY: principal identificador; única por tabela; não aceita NULL.
    • UNIQUE: garante unicidade; pode haver várias por tabela; pode aceitar NULL conforme SGBD.

    Chave candidata x chave primária#

    • toda chave primária é candidata;
    • nem toda chave candidata é primária.

    Integridade de entidade x referencial#

    • entidade: garante identificação da própria linha.
    • referencial: garante coerência entre tabelas.

    NULL x valor vazio#

    • NULL = ausência/desconhecimento de valor.
    • '' = string vazia.
    • 0 = valor numérico válido.

    Consistência x integridade#

    • integridade: conformidade com restrições definidas.
    • consistência: estado válido do banco segundo regras e transações; em provas, os termos podem aparecer próximos, mas não são sinônimos perfeitos.

    5. Exemplos e Aplicações#

    Exemplo de chave composta#

    CREATE TABLE Matricula (
    id_aluno INT,
    id_disciplina INT,
    data_matricula DATE NOT NULL,
    PRIMARY KEY (id_aluno, id_disciplina)
);

    Aqui, a combinação dos dois atributos identifica a linha.

    Exemplo de violação referencial#

    Se Empregado.id_departamento = 10, então deve existir Departamento.id_departamento = 10, salvo se a FK permitir NULL e o valor for nulo.

    Cenário real#

    Em sistema bancário:

    • conta deve ter identificador único → PRIMARY KEY
    • CPF do cliente pode ser único → UNIQUE
    • saldo não pode ser negativo, se regra exigir → CHECK
    • agência da conta deve existir → FOREIGN KEY

    6. Pegadinhas de Prova#

    • “Toda chave única é chave primária.” Errado.
    • “Uma tabela pode ter várias chaves primárias.” Errado.
    • “Chave estrangeira sempre referencia chave primária.” Errado; pode referenciar atributo(s) UNIQUE, conforme SGBD/modelagem.
    • “Integridade referencial impede valores nulos em chave estrangeira.” Errado; pode permitir NULL.
    • “PRIMARY KEY é apenas uma restrição de unicidade.” Errado; também impõe não nulidade.
    • “CHECK substitui FOREIGN KEY.” Errado.
    • “Normalização garante sozinha integridade referencial.” Errado.
    • Atenção a termos absolutos: sempre, nunca, somente, exclusivamente.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    PRIMARY KEYÚnica e não nulaNão pode haver mais de uma por tabela
    UNIQUEImpede duplicidadeNão se confunde com chave primária
    FOREIGN KEYReferencia valor existente em outra tabelaPode não ser única e pode ser composta
    NOT NULLProíbe valor nuloNulo não é igual a vazio
    CHECKImpõe condição lógicaNão substitui integridade referencial
    Chave candidataSuperchave mínimaPode não ser a escolhida como primária
    Integridade de entidadePK deve identificar cada tuplaRelaciona-se à própria tabela
    Integridade referencialFK deve apontar para valor válidoPode admitir NULL, se permitido

    8. Simulado (Certo ou Errado)#

    1. A integridade de entidade determina que nenhum atributo de uma tabela possa conter valores nulos.
    2. Uma chave primária é, necessariamente, única e não admite valores nulos.
    3. Uma tabela relacional pode possuir mais de uma chave primária, desde que todas sejam candidatas.
    4. A integridade referencial exige que todo valor de chave estrangeira corresponda a um valor existente na tabela referenciada, salvo quando o valor nulo for permitido.
    5. Toda chave estrangeira deve referenciar exclusivamente a chave primária da tabela pai.
    6. A restrição UNIQUE e a PRIMARY KEY são equivalentes em todos os aspectos.
    7. Uma chave composta é formada por dois ou mais atributos e pode ser chave primária ou estrangeira.
    8. A normalização reduz redundâncias e anomalias, mas não elimina a necessidade de restrições explícitas de integridade no banco.

    Gabarito comentado#

    1. Errado.
    Pegadinha: integridade de entidade refere-se à chave primária, não a todos os atributos.

    2. Certo.
    PRIMARY KEY implica unicidade e não nulidade.

    3. Errado.
    Só pode existir uma chave primária por tabela, embora possam existir várias chaves candidatas.

    4. Certo.
    Essa é a definição clássica de integridade referencial.

    5. Errado.
    Pegadinha clássica: pode referenciar também chave com restrição UNIQUE, conforme implementação/modelagem.

    6. Errado.
    PRIMARY KEY é identificador principal e não aceita NULL; UNIQUE não é idêntica a ela.

    7. Certo.
    Tanto PK quanto FK podem ser compostas.

    8. Certo.
    Normalização organiza o esquema, mas as restrições continuam necessárias para garantir integridade operacional.
