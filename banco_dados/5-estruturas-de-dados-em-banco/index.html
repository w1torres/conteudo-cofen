    5. Estruturas de dados em banco#

    1. Fundamentos e Funcionamento#

    Em banco de dados relacional, estruturas de dados são os objetos usados para armazenar, organizar, acessar e relacionar informações. No contexto de prova, os principais são: tabelas, views, índices, chaves e relacionamentos.

    Definição técnica#

    • Tabela: estrutura física/lógica composta por linhas (tuplas/registros) e colunas (atributos/campos).
    • View: tabela virtual definida por uma consulta SQL; em regra, não armazena dados próprios, mas apresenta dados derivados de tabelas base.
    • Índice: estrutura auxiliar de acesso que acelera buscas, ordenações e junções, à custa de espaço e custo adicional em inserções/atualizações/exclusões.
    • Chave: atributo ou conjunto de atributos usado para identificar registros ou estabelecer integridade.
    • Relacionamento: associação entre entidades/tabelas, implementada no modelo relacional por chaves primárias e estrangeiras.

    Como funciona internamente#

    No ciclo de modelagem:

    • Modelo conceitual: entidades, atributos e relacionamentos.
    • Modelo lógico: transformação em tabelas, colunas, chaves e restrições.
    • Modelo físico: tipos de dados, índices, particionamento, armazenamento.

    A tabela armazena dados; a view abstrai consultas; o índice otimiza acesso; as chaves garantem unicidade e integridade; os relacionamentos materializam vínculos entre tabelas.

    Componentes principais#

    • Colunas, tipos de dados, restrições (NOT NULL, UNIQUE, CHECK, DEFAULT)
    • Chave primária e estrangeira
    • Índices primários/secundários, únicos/não únicos
    • Views simples e complexas
    • Cardinalidades: 1:1, 1:N, N:N

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Tabelas#

    • Representam relações no modelo relacional.
    • Possuem esquema (estrutura) e instâncias (dados).
    • Cada coluna possui domínio/tipo.
    • Linhas não devem ser duplicadas conceitualmente; na prática, sem chave, o SGBD pode permitir duplicidade.
    • Ordem das linhas é irrelevante no modelo relacional.
    • Ordem das colunas também não é relevante logicamente.

    Views#

    • São definidas por SELECT.
    • Em regra, são virtuais.
    • Podem simplificar segurança e abstração.
    • Nem toda view é atualizável.
    • View materializada armazena resultado fisicamente (quando suportada pelo SGBD), diferindo da view comum.

    Índices#

    • Melhoram desempenho de leitura.
    • Pioram custo de escrita.
    • Podem ser criados sobre uma ou mais colunas.
    • Índice único impõe unicidade.
    • Nem toda chave gera automaticamente índice em todos os SGBDs, embora seja comum.
    • Índice não altera o resultado lógico da consulta, apenas o caminho de acesso.

    Chaves#

    • Superchave: conjunto que identifica unicamente.
    • Chave candidata: superchave mínima.
    • Chave primária (PK): candidata escolhida.
    • Chave alternativa: candidata não escolhida como primária.
    • Chave estrangeira (FK): referencia PK ou coluna(s) UNIQUE de outra tabela.
    • PK não admite NULL.
    • FK pode admitir NULL, salvo restrição contrária.

    Relacionamentos#

    • 1:1: FK com unicidade.
    • 1:N: FK no lado N.
    • N:N: exige tabela associativa/intermediária.
    • Integridade referencial impede referência inválida.
    • Ações comuns: CASCADE, SET NULL, RESTRICT, NO ACTION.

    Normalização#

    • 1FN: atributos atômicos, sem grupos repetitivos.
    • 2FN: eliminar dependência parcial de chave composta.
    • 3FN: eliminar dependência transitiva de atributo não chave.
    • BCNF: todo determinante deve ser chave candidata.

    SQL e comportamento#

    • PRIMARY KEY = unicidade + não nulidade.
    • UNIQUE permite, em muitos SGBDs, NULL (pegadinha clássica).
    • FOREIGN KEY garante consistência entre tabelas.
    • JOIN implementa consulta sobre relacionamentos.

    ETL/ELT#

    • Estruturas de banco participam da carga.
    • Índices excessivos prejudicam carga massiva.
    • Views podem servir como camada semântica para consumo analítico.

    3. Regras Técnicas e Comportamento#

    Tabelas#

    Exemplo:

    CREATE TABLE cliente (
  id_cliente INT PRIMARY KEY,
  nome VARCHAR(100) NOT NULL,
  cpf CHAR(11) UNIQUE,
  data_cadastro DATE DEFAULT CURRENT_DATE
);

    Regras:

    • PK identifica unicamente cada linha.
    • UNIQUE evita duplicidade, mas não equivale sempre a PK.
    • Tipos de dados influenciam armazenamento e comparação.

    Views#

    CREATE VIEW vw_clientes_ativos AS
SELECT id_cliente, nome
FROM cliente
WHERE ativo = 'S';
    • View simples sobre uma tabela, sem agregação, tende a ser atualizável.
    • Views com GROUP BY, agregações, DISTINCT, UNION ou junções complexas geralmente não são atualizáveis.

    Índices#

    CREATE INDEX idx_cliente_nome ON cliente(nome);
CREATE UNIQUE INDEX idx_cliente_cpf ON cliente(cpf);
    • Úteis em colunas de busca, junção, ordenação e filtro.
    • Pouco úteis em colunas com baixa seletividade, dependendo do cenário.
    • Índices compostos obedecem ordem das colunas; isso afeta aproveitamento.

    Chaves e relacionamentos#

    CREATE TABLE pedido (
  id_pedido INT PRIMARY KEY,
  id_cliente INT NOT NULL,
  valor NUMERIC(10,2),
  CONSTRAINT fk_pedido_cliente
    FOREIGN KEY (id_cliente) REFERENCES cliente(id_cliente)
);
    • Em 1:N, vários pedidos podem referenciar um cliente.
    • FK não precisa ser única no lado filho.
    • Integridade referencial impede inserir pedido com cliente inexistente.

    N:N#

    CREATE TABLE aluno_disciplina (
  id_aluno INT,
  id_disciplina INT,
  PRIMARY KEY (id_aluno, id_disciplina),
  FOREIGN KEY (id_aluno) REFERENCES aluno(id_aluno),
  FOREIGN KEY (id_disciplina) REFERENCES disciplina(id_disciplina)
);
    • Tabela associativa resolve relacionamento muitos-para-muitos.
    • Pode conter atributos próprios, como data_matricula.

    4. Diferenças Críticas#

    Tabela x View#

    • Tabela armazena dados.
    • View normalmente não armazena; apenas apresenta consulta.
    • View materializada pode armazenar resultado.

    PK x UNIQUE#

    • PK: única por tabela (conceitualmente), não nula.
    • UNIQUE: podem existir várias por tabela; em muitos SGBDs aceita NULL.

    PK x FK#

    • PK identifica a própria linha.
    • FK referencia linha de outra tabela (ou da própria tabela, em autorrelacionamento).

    Índice x Chave#

    • Chave é conceito de integridade/modelagem.
    • Índice é estrutura de desempenho.
    • Uma chave pode gerar índice, mas não são sinônimos.

    Relacionamento lógico x implementação física#

    • No modelo conceitual há relacionamento entre entidades.
    • No modelo lógico/físico isso se implementa com FK e tabelas associativas.

    5. Exemplos e Aplicações#

    Modelagem#

    Conceitual: Cliente realiza Pedido.
    Lógico:

    • CLIENTE(id_cliente, nome, cpf)
    • PEDIDO(id_pedido, data, valor, id_cliente)

    Físico:

    • Definição de tipos, PK, FK, índices.

    Consulta com relacionamento#

    SELECT c.nome, p.id_pedido, p.valor
FROM cliente c
JOIN pedido p ON p.id_cliente = c.id_cliente;

    Uso de view para segurança#

    CREATE VIEW vw_cliente_publico AS
SELECT id_cliente, nome
FROM cliente;
    • Oculta colunas sensíveis, como CPF.

    ETL/ELT#

    Em carga de data warehouse:

    • remover/desabilitar índices pode acelerar carga;
    • recriar índices após carga pode ser mais eficiente;
    • views podem padronizar acesso analítico.

    6. Pegadinhas de Prova#

    • “Toda view armazena dados fisicamente.” → Errado.
    • “Índice sempre melhora o desempenho geral do banco.” → Errado; melhora leitura, mas pode piorar escrita.
    • “Chave estrangeira deve ser sempre única.” → Errado.
    • “Relacionamento N:N é implementado diretamente por uma única FK.” → Errado.
    • “UNIQUE e PRIMARY KEY são equivalentes.” → Errado.
    • “A ordem das linhas em uma tabela relacional é relevante logicamente.” → Errado.
    • “Toda view é atualizável.” → Errado.
    • “Toda chave primária é uma chave candidata, mas nem toda chave candidata é primária.” → Certo.

    Termos que invalidam questões:

    • sempre, nunca, somente, obrigatoriamente, equivale a.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    TabelaArmazena dados em linhas e colunasOrdem das linhas não é lógica
    ViewConsulta armazenada, geralmente virtualNem toda view é atualizável
    View materializadaArmazena resultado fisicamenteNão é sinônimo de view comum
    ÍndiceAcelera leitura/acessoPode degradar inserção e atualização
    PKIdentifica unicamente e não aceita NULLNão se confunde com índice
    FKReferencia PK/UNIQUE de outra tabelaPode aceitar NULL
    UNIQUEImpõe unicidadeEm muitos SGBDs aceita NULL
    1:NFK no lado NNão se coloca FK no lado 1
    N:NExige tabela associativaNão é resolvido por uma única coluna
    3FNRemove dependência transitivaNão trata apenas repetição de grupos

    8. Simulado (Certo ou Errado)#

    1. A chave primária de uma tabela pode assumir valor nulo, desde que não haja duplicidade.
    2. Uma view é, por definição, uma estrutura física que armazena permanentemente os dados resultantes de uma consulta.
    3. Em um relacionamento um-para-muitos, a chave estrangeira deve ficar na tabela do lado muitos.
    4. Índices melhoram consultas de leitura, mas podem aumentar o custo de operações de escrita.
    5. Um relacionamento muitos-para-muitos, no modelo relacional, é normalmente implementado por uma tabela associativa.
    6. Toda restrição UNIQUE é equivalente a uma PRIMARY KEY.
    7. Uma chave estrangeira pode referenciar uma chave primária da própria tabela, caracterizando autorrelacionamento.
    8. Na modelagem relacional, a ordem física das linhas de uma tabela é parte essencial do significado lógico dos dados.

    Gabarito comentado#

    1. Errado. PK não admite NULL. Pegadinha: confundir com UNIQUE.
    2. Errado. View comum é virtual; quem pode armazenar resultado é a view materializada.
    3. Certo. Em 1:N, a FK fica no lado N, apontando para o lado 1.
    4. Certo. Índice acelera leitura, mas exige manutenção em INSERT, UPDATE, DELETE.
    5. Certo. A implementação padrão é por tabela intermediária com FKs.
    6. Errado. PRIMARY KEY implica não nulidade; UNIQUE não é necessariamente igual e pode haver várias por tabela.
    7. Certo. É possível FK autorreferente, como empregado referenciando gerente.
    8. Errado. No modelo relacional, ordem de linhas não compõe o significado lógico.
