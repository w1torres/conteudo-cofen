    5. Estruturas de dados em banco#

    1. Fundamentos e Funcionamento#

    As estruturas de dados em banco de dados são fundamentais para o armazenamento, organização, manipulação e recuperação eficiente de dados. Entre as principais estruturas, destacam-se tabelas, views, índices, chaves e relacionamentos. Internamente, essas estruturas permitem que os sistemas de gerenciamento de banco de dados (SGBDs) otimizem o acesso aos dados e garantam a integridade e a consistência das informações.

    2. Estrutura Completa do Tema (Nível Prova)#

    • Tabelas: Coleções de dados organizados em linhas e colunas, onde cada coluna representa um atributo e cada linha, um registro.
    • Views: Representações virtuais de uma ou mais tabelas. São consultas armazenadas que podem ser utilizadas como se fossem tabelas.
    • Índices: Estruturas que melhoram a velocidade de recuperação dos dados, criando um caminho de acesso rápido para os registros de uma tabela.
    • Chaves:
      • Primária (PK): Identifica de forma única cada registro na tabela.
      • Estrangeira (FK): Estabelece um link entre duas tabelas, referenciando a chave primária de outra tabela.
    • Relacionamentos: Descrevem como as tabelas se conectam entre si, podendo ser um-para-um, um-para-muitos ou muitos-para-muitos.

    3. Regras Técnicas e Comportamento#

    • Tabelas: Devem ser normalizadas para evitar redundâncias e dependências.
    • Views: Podem ser atualizáveis ou somente de leitura, dependendo da complexidade da consulta SQL que as define.
    • Índices: A criação de índices deve ser balanceada, pois, embora melhorem a leitura, podem tornar a escrita mais lenta.
    • Chaves Primárias: Não podem conter valores nulos e devem ser únicas em cada tabela.
    • Chaves Estrangeiras: Garantem a integridade referencial entre as tabelas.
    • Relacionamentos: A integridade referencial deve ser mantida, assegurando que as chaves estrangeiras correspondam sempre a uma chave primária válida.

    4. Diferenças Críticas#

    • Tabela vs. View: Tabelas armazenam dados fisicamente, enquanto views são apenas representações lógicas de uma ou mais tabelas.
    • Índice Clusterizado vs. Não Clusterizado: Índices clusterizados armazenam fisicamente as linhas da tabela na ordem do índice, enquanto os não clusterizados mantêm um ponteiro para os dados.
    • Chave Primária vs. Chave Estrangeira: Chaves primárias identificam unicamente registros dentro de sua própria tabela, enquanto chaves estrangeiras referenciam chaves primárias em outras tabelas.

    5. Exemplos e Aplicações#

    -- Criação de tabela
CREATE TABLE Alunos (
    AlunoID INT PRIMARY KEY,
    Nome VARCHAR(100),
    DataNascimento DATE
);

-- Criação de view
CREATE VIEW Vw_AlunosAtivos AS
SELECT Nome, DataNascimento
FROM Alunos
WHERE Ativo = 1;

-- Criação de índice
CREATE INDEX idx_nome ON Alunos(Nome);

-- Estabelecendo um relacionamento
ALTER TABLE Matriculas
ADD CONSTRAINT FK_AlunoID
FOREIGN KEY (AlunoID) REFERENCES Alunos(AlunoID);

    6. Pegadinhas de Prova#

    • Afirmar que views armazenam dados fisicamente.
    • Confundir índice clusterizado com não clusterizado, especialmente em relação ao armazenamento de dados.
    • Dizer que uma chave estrangeira pode referenciar qualquer coluna, não necessariamente uma chave primária ou única.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    TabelasArmazenam dados fisicamente em linhas e colunas.“Views também armazenam dados.”
    ViewsSão consultas armazenadas que representam dados de uma ou mais tabelas.“Views podem ser atualizadas sempre.”
    ÍndicesMelhoram a performance de leitura; podem ser clusterizados ou não.“Índices não afetam a performance de escrita.”
    Chaves PrimáriasIdentificam unicamente cada registro na tabela.“Chaves primárias podem ser nulas.”
    Chaves EstrangeirasReferenciam a chave primária de outra tabela.“Podem referenciar qualquer coluna.”

    8. Simulado (Certo ou Errado)#

    1. Uma view pode ser utilizada para esconder a complexidade de consultas envolvendo múltiplas tabelas. (CERTO)
    2. Índices não clusterizados armazenam os dados fisicamente na ordem do índice. (ERRADO)
    3. Chaves estrangeiras podem ser definidas em mais de uma coluna para estabelecer um relacionamento. (CERTO)
    4. Tabelas não normalizadas são preferíveis para melhorar a performance de consultas. (ERRADO)
    5. Uma chave primária pode ser composta por mais de uma coluna para identificar unicamente um registro. (CERTO)
    6. Views atualizáveis não podem incluir joins de múltiplas tabelas. (ERRADO)
    7. Índices melhoram a performance de leitura, mas podem comprometer a de escrita. (CERTO)
    8. Uma chave estrangeira deve sempre referenciar uma chave primária ou uma chave única. (CERTO)

    Gabarito comentado:#

    1. CERTO: Views são frequentemente usadas para simplificar consultas complexas e para implementar uma camada de abstração sobre a estrutura física dos dados.
    2. ERRADO: Índices não clusterizados criam uma estrutura separada que aponta para os dados, ao invés de armazená-los fisicamente ordenados.
    3. CERTO: Chaves estrangeiras compostas são utilizadas quando é necessário referenciar uma chave primária composta em outra tabela.
    4. ERRADO: A normalização é essencial para evitar redundâncias e dependências, melhorando a integridade e a manutenção dos dados.
    5. CERTO: Chaves primárias compostas são usadas quando nenhum atributo único identifica registros de forma única, mas uma combinação deles sim.
    6. ERRADO: Views atualizáveis podem incluir joins, mas há restrições específicas que determinam se uma view baseada em joins é atualizável.
    7. CERTO: Enquanto índices aceleram a recuperação de dados, eles podem tornar as operações de inserção, atualização e exclusão mais lentas devido à necessidade de manter o índice atualizado.
    8. CERTO: A integridade referencial é mantida por meio de chaves estrangeiras que apontam para chaves primárias ou chaves únicas, garantindo relações consistentes entre tabelas.
