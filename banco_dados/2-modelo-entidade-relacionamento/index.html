    2. Modelo Entidade-Relacionamento#

    1. Fundamentos e Funcionamento#

    Definição técnica#

    O Modelo Entidade-Relacionamento (MER) é um modelo conceitual de dados usado para representar, de forma abstrata, os dados de um domínio e as associações entre eles. Seu produto gráfico é o Diagrama Entidade-Relacionamento (DER).

    Ele serve para responder, antes da implementação física:

    • quais objetos do negócio existem;
    • quais características possuem;
    • como se relacionam;
    • quais restrições devem ser respeitadas.

    Como funciona internamente#

    O MER abstrai a realidade em:

    • Entidades: objetos ou conceitos relevantes.
    • Atributos: propriedades das entidades ou relacionamentos.
    • Relacionamentos: associações entre entidades.
    • Restrições: cardinalidade, participação, identificação, especialização/generalização.

    Componentes principais#

    Entidade#

    Conjunto de objetos com características comuns.
    Ex.: Aluno, Curso, Departamento.

    Instância de entidade#

    Ocorrência específica da entidade.
    Ex.: Aluno = João, matrícula 123.

    Atributo#

    Característica da entidade/relacionamento. Pode ser:

    • simples (atômico);
    • composto;
    • monovalorado;
    • multivalorado;
    • derivado;
    • identificador (chave).

    Relacionamento#

    Associação entre entidades.
    Ex.: Aluno cursa Disciplina.

    Grau do relacionamento#

    Número de entidades participantes:

    • unário/recursivo;
    • binário;
    • ternário;
    • n-ário.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de prova#

    Modelagem de dados#

    • Conceitual: independente de SGBD; usa MER/DER.
    • Lógica: traduz o modelo conceitual para o modelo lógico (geralmente relacional).
    • Física: define armazenamento, índices, tipos físicos, particionamento etc.

    Entidades#

    • Forte: possui identificador próprio.
    • Fraca: depende de outra entidade para identificação.
    • Entidade fraca participa de relacionamento identificador.

    Atributos#

    • Simples: indivisível.
    • Composto: pode ser decomposto. Ex.: endereço.
    • Monovalorado: um valor por ocorrência.
    • Multivalorado: vários valores por ocorrência. Ex.: telefones.
    • Derivado: calculado a partir de outro. Ex.: idade derivada de data_nascimento.
    • Chave: identifica univocamente a entidade.

    Chaves#

    • Superchave: conjunto que identifica univocamente.
    • Chave candidata: superchave mínima.
    • Chave primária: candidata escolhida.
    • Chave alternativa: candidata não escolhida.
    • Chave estrangeira: no modelo relacional, referencia chave de outra tabela.

    Relacionamentos#

    • 1:1
    • 1:N
    • N:N
    • Recursivo
    • Ternário e superiores

    Restrições estruturais#

    • Cardinalidade máxima: 1 ou N.
    • Participação mínima: opcional (0) ou total/obrigatória (1).
    • Notação comum: (min,max).

    Especialização/Generalização#

    • Generalização: abstrai características comuns.
    • Especialização: cria subtipos.
    • Disjunta: uma instância pertence a um único subtipo.
    • Sobreposta: pode pertencer a vários subtipos.
    • Total: toda instância do supertipo pertence a algum subtipo.
    • Parcial: nem toda instância precisa pertencer.

    Agregação#

    • Trata um relacionamento como entidade de nível superior para participar de outro relacionamento.

    3. Regras Técnicas e Comportamento#

    Cardinalidade e participação#

    A banca costuma confundir:

    • Cardinalidade = quantidade de ocorrências associadas.
    • Participação = obrigatoriedade da associação.

    Exemplo:

    • Um Departamento possui 0..N empregados.
    • Um Empregado pertence a 1..1 departamento.

    Logo:

    • Departamento → participação parcial no relacionamento.
    • Empregado → participação total.

    Entidade fraca#

    Características:

    • não possui chave completa própria;
    • depende da entidade forte;
    • possui chave parcial;
    • sua identificação ocorre pela combinação: chave da entidade forte + chave parcial.

    Ex.: Dependente de Empregado.
    Dependente pode ser identificado por (matricula_empregado, nome_dependente).

    Relacionamento N:N#

    No MER é perfeitamente válido.
    No modelo relacional, deve ser transformado em tabela associativa.

    Ex.:

    • Aluno — cursa — Disciplina (N:N)

    Modelo lógico:

    CREATE TABLE Aluno (
  id_aluno INT PRIMARY KEY,
  nome VARCHAR(100)
);

CREATE TABLE Disciplina (
  id_disciplina INT PRIMARY KEY,
  nome VARCHAR(100)
);

CREATE TABLE Matricula (
  id_aluno INT,
  id_disciplina INT,
  data_matricula DATE,
  PRIMARY KEY (id_aluno, id_disciplina),
  FOREIGN KEY (id_aluno) REFERENCES Aluno(id_aluno),
  FOREIGN KEY (id_disciplina) REFERENCES Disciplina(id_disciplina)
);

    Atributo multivalorado#

    No MER pode existir.
    No modelo relacional, normalmente deve ser decomposto em outra relação para evitar grupos repetitivos e violação da 1FN.

    Atributo derivado#

    Pode ser representado no modelo conceitual, mas nem sempre deve ser armazenado fisicamente, pois pode gerar redundância e inconsistência.

    Normalização e relação com MER#

    Embora normalização seja mais ligada ao modelo relacional, o MER bem construído já reduz redundâncias. Em prova:

    • MER = abstração conceitual;
    • normalização = refinamento lógico.

    Regras formais lembradas:

    • 1FN: atributos atômicos, sem grupos repetitivos.
    • 2FN: 1FN + dependência total da chave.
    • 3FN: 2FN + sem dependência transitiva de atributo não-chave.

    4. Diferenças Críticas#

    MER x DER#

    • MER: modelo conceitual.
    • DER: representação gráfica do MER.

    Modelo conceitual x lógico x físico#

    • Conceitual: visão do negócio, sem foco em implementação.
    • Lógico: estrutura em tabelas, colunas, chaves.
    • Físico: detalhes de armazenamento no SGBD.

    Entidade x tabela#

    • Entidade não é automaticamente tabela, embora frequentemente seja convertida em uma.
    • Relacionamentos N:N e atributos multivalorados também podem virar tabelas.

    Atributo composto x multivalorado#

    • Composto: um valor com partes internas. Ex.: endereço.
    • Multivalorado: vários valores do mesmo atributo. Ex.: telefones.

    Cardinalidade x opcionalidade#

    • Cardinalidade máxima: 1 ou N.
    • Opcionalidade/participação mínima: 0 ou 1.

    5. Exemplos e Aplicações#

    Exemplo conceitual#

    Sistema acadêmico:

    • Aluno(matrícula, nome, data_nascimento)
    • Curso(cod_curso, nome)
    • Disciplina(cod_disciplina, nome, carga_horaria)

    Relacionamentos:

    • Aluno matricula-se em Disciplina → N:N
    • Curso oferece Disciplina → 1:N
    • Aluno pertence a Curso → N:1

    Exemplo de atributo derivado#

    • data_nascimento
    • idade = derivada de data_nascimento

    Exemplo de entidade fraca#

    • Pedido e ItemPedido
    • ItemPedido depende de Pedido para existir e ser identificado.

    ETL/ELT e modelagem#

    Em ambientes analíticos:

    • o MER costuma apoiar a compreensão do domínio;
    • no DW, a modelagem pode migrar para dimensional (fato e dimensão), que não substitui o MER, mas atende outro objetivo.

    6. Pegadinhas de Prova#

    • Dizer que MER é modelo lógico → errado.
    • Dizer que DER e MER são sinônimos absolutos → errado; DER representa o MER.
    • Dizer que todo relacionamento vira tabela → errado; em geral, 1:N pode ser resolvido com chave estrangeira.
    • Dizer que atributo derivado deve sempre ser armazenado → errado.
    • Dizer que entidade fraca não possui nenhum identificador → errado; possui chave parcial.
    • Dizer que cardinalidade e participação são a mesma coisa → errado.
    • Dizer que relacionamento N:N não pode existir no MER → errado; a restrição é na implementação relacional direta.
    • Termos absolutos como sempre, nunca, somente costumam invalidar assertivas.

    7. Tabela de Fixação#

    ConceitoRegraPegadinha
    MERModelo conceitualConfundir com modelo lógico
    DERDiagrama do MERTratar como o próprio modelo em sentido estrito
    Entidade fracaDepende de entidade forte para identificaçãoAchar que não tem chave parcial
    Atributo derivadoPode ser calculadoSupor armazenamento obrigatório
    Atributo multivaloradoPode ter vários valoresConfundir com atributo composto
    Relacionamento N:NVálido no conceitualNo relacional vira tabela associativa
    Participação totalMínimo 1Confundir com cardinalidade máxima 1
    GeneralizaçãoSupertipo para subtiposConfundir com agregação

    8. Simulado (Certo ou Errado)#

    1. O MER é utilizado na modelagem conceitual e independe, em regra, do SGBD que implementará o banco.
    2. No DER, um atributo multivalorado é, por definição, indivisível e atômico.
    3. Uma entidade fraca pode depender de uma entidade forte para sua identificação.
    4. Todo relacionamento identificado no MER deve ser transformado em tabela no modelo relacional.
    5. Um relacionamento muitos-para-muitos, ao ser mapeado para o modelo relacional, normalmente origina uma tabela associativa.
    6. Cardinalidade e participação são conceitos equivalentes e intercambiáveis no MER.
    7. Um atributo derivado, como idade calculada a partir da data de nascimento, pode ser representado no modelo conceitual sem necessidade de armazenamento físico obrigatório.
    8. Na especialização total, toda ocorrência do supertipo deve pertencer a pelo menos um subtipo.

    Gabarito comentado#

    1. Certo.
    Pegadinha: confundir MER com modelo lógico/físico.

    2. Errado.
    Atributo multivalorado não significa atômico; ele admite múltiplos valores.

    3. Certo.
    Essa é a essência da entidade fraca.

    4. Errado.
    Pegadinha clássica. Nem todo relacionamento vira tabela; muitos 1:N viram chave estrangeira.

    5. Certo.
    Mapeamento padrão do N:N para o relacional.

    6. Errado.
    Cardinalidade trata da quantidade; participação, da obrigatoriedade.

    7. Certo.
    Atributo derivado pode existir conceitualmente sem ser persistido.

    8. Certo.
    Na especialização total, nenhuma instância do supertipo fica fora dos subtipos.
