    4. SQL#

    1. Fundamentos e Funcionamento#

    SQL (Structured Query Language) é a linguagem padrão para definição, manipulação, consulta e controle de dados em bancos relacionais. Embora seja chamada de “linguagem”, ela é declarativa em grande parte: o usuário informa o que deseja, e o SGBD decide como executar.

    Como funciona internamente#

    Quando um comando SQL é enviado ao SGBD, em geral ocorre:

    1. Parsing: análise léxica e sintática.
    2. Validação semântica: verificação de tabelas, colunas, tipos, permissões.
    3. Otimização: escolha do plano de execução.
    4. Execução: leitura/escrita em memória e disco.
    5. Retorno: conjunto de resultados ou confirmação da operação.

    Componentes principais#

    SQL costuma ser dividido em sublinguagens:

    • DDL (Data Definition Language): define estruturas.
      • CREATE, ALTER, DROP, TRUNCATE (muitas bancas incluem aqui)
    • DML (Data Manipulation Language): manipula dados.
      • INSERT, UPDATE, DELETE
    • DQL (Data Query Language): consulta dados.
      • SELECT
    • DCL (Data Control Language): controle de privilégios.
      • GRANT, REVOKE
    • TCL (Transaction Control Language): controle transacional.
      • COMMIT, ROLLBACK, SAVEPOINT

    Atenção de prova: algumas classificações variam por autor/SGBD. A banca pode considerar SELECT como parte da DML ou separadamente como DQL.

    2. Estrutura Completa do Tema (Nível Prova)#

    Checklist de SQL#

    • SQL é linguagem padrão para bancos relacionais.
    • É predominantemente declarativa.
    • Atua sobre esquema e dados.
    • Pode ser classificada em DDL, DML, DQL, DCL e TCL.
    • CREATE, ALTER, DROP alteram estrutura.
    • INSERT, UPDATE, DELETE alteram conteúdo.
    • SELECT consulta dados.
    • WHERE filtra linhas antes do agrupamento.
    • HAVING filtra grupos após GROUP BY.
    • ORDER BY ordena resultado final.
    • DISTINCT elimina duplicidades no resultado.
    • JOIN combina linhas de tabelas.
    • INNER JOIN retorna apenas correspondências.
    • LEFT JOIN preserva todas as linhas da tabela à esquerda.
    • RIGHT JOIN preserva as da direita.
    • FULL OUTER JOIN preserva todas de ambas.
    • NULL não é zero, nem vazio, nem falso.
    • Comparações com NULL usam IS NULL / IS NOT NULL.
    • Agregações: COUNT, SUM, AVG, MIN, MAX.
    • COUNT(*) conta linhas, inclusive com NULL.
    • COUNT(coluna) ignora NULL na coluna.
    • Restrições: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK, DEFAULT.
    • PRIMARY KEY implica unicidade e não nulidade.
    • Uma tabela tem apenas uma chave primária, podendo ser composta.
    • FOREIGN KEY implementa integridade referencial.
    • Transações obedecem propriedades ACID.
    • COMMIT confirma; ROLLBACK desfaz alterações não confirmadas.
    • TRUNCATE remove todas as linhas, em geral sem WHERE.
    • DELETE permite WHERE.
    • DROP remove o objeto.
    • Visões (VIEW) armazenam consulta, não necessariamente dados.
    • Índices aceleram leitura, mas podem degradar escrita.

    Relação com modelagem#

    • Conceitual: entidades e relacionamentos.
    • Lógica: tabelas, atributos, chaves.
    • Física: tipos, índices, particionamento, armazenamento.

    Normalização ligada ao SQL#

    • 1FN: atributos atômicos, sem grupos repetitivos.
    • 2FN: eliminar dependência parcial de chave composta.
    • 3FN: eliminar dependência transitiva.
    • BCNF: todo determinante deve ser chave candidata.

    3. Regras Técnicas e Comportamento#

    Ordem lógica de processamento de uma consulta#

    A banca gosta de confundir ordem escrita com ordem lógica:

    1. FROM
    2. JOIN
    3. WHERE
    4. GROUP BY
    5. HAVING
    6. SELECT
    7. DISTINCT
    8. ORDER BY

    Regras importantes#

    • WHERE não usa funções de agregação diretamente; para isso, usa-se HAVING.
    • GROUP BY exige que colunas no SELECT sejam:
      • agregadas, ou
      • incluídas no agrupamento.
    • ORDER BY pode usar alias do SELECT em muitos SGBDs.
    • INNER JOIN equivale a interseção lógica entre conjuntos relacionados.
    • OUTER JOIN inclui não correspondentes com preenchimento por NULL.

    Integridade#

    • Entidade: chave primária não pode ser NULL.
    • Referencial: chave estrangeira deve referenciar valor existente (ou ser NULL, se permitido).
    • Domínio: tipo, formato e restrições válidas.

    Transações#

    Propriedades ACID:

    • Atomicidade: tudo ou nada.
    • Consistência: preserva regras do banco.
    • Isolamento: transações não interferem indevidamente.
    • Durabilidade: após COMMIT, persistência garantida.

    Limitações e condições#

    • SQL padrão existe, mas cada SGBD possui extensões.
    • Nem todo comando DDL é totalmente transacional em todos os bancos.
    • NULL gera lógica ternária: verdadeiro, falso, desconhecido.

    4. Diferenças Críticas#

    DELETE x TRUNCATE x DROP#

    • DELETE
      • remove linhas
      • pode usar WHERE
      • operação DML
    • TRUNCATE
      • remove todas as linhas
      • normalmente sem WHERE
      • geralmente DDL em muitas classificações
      • mais rápido em muitos cenários
    • DROP
      • remove o objeto inteiro (tabela, visão etc.)

    WHERE x HAVING#

    • WHERE: filtra linhas antes do agrupamento.
    • HAVING: filtra grupos após agregação.

    PRIMARY KEY x UNIQUE#

    • PRIMARY KEY
      • única por tabela
      • não aceita NULL
    • UNIQUE
      • pode haver várias por tabela
      • em muitos SGBDs aceita NULL (pegadinha: isso pode variar)

    INNER JOIN x OUTER JOIN#

    • INNER: só correspondentes.
    • LEFT/RIGHT/FULL OUTER: preservam linhas sem correspondência.

    COUNT(*) x COUNT(coluna)#

    • COUNT(*): conta todas as linhas.
    • COUNT(coluna): ignora linhas em que a coluna é NULL.

    5. Exemplos e Aplicações#

    Criação de tabela#

    CREATE TABLE Departamento (
    id_departamento INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL UNIQUE
);

CREATE TABLE Empregado (
    id_empregado INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    salario DECIMAL(10,2) CHECK (salario >= 0),
    id_departamento INT,
    CONSTRAINT fk_emp_dep
        FOREIGN KEY (id_departamento)
        REFERENCES Departamento(id_departamento)
);

    Consulta com filtro, agrupamento e ordenação#

    SELECT id_departamento, COUNT(*) AS qtd, AVG(salario) AS media
FROM Empregado
WHERE salario > 2000
GROUP BY id_departamento
HAVING COUNT(*) >= 2
ORDER BY media DESC;

    Junção#

    SELECT e.nome, d.nome AS departamento
FROM Empregado e
LEFT JOIN Departamento d
    ON e.id_departamento = d.id_departamento;

    Transação#

    BEGIN TRANSACTION;

UPDATE Empregado
SET salario = salario * 1.10
WHERE id_departamento = 1;

ROLLBACK;

    ETL/ELT e SQL#

    Em processos de dados:

    • ETL: extrai, transforma antes, carrega depois.
    • ELT: extrai, carrega primeiro, transforma no destino. SQL é muito usado na etapa de transformação, limpeza, agregação e carga.

    6. Pegadinhas de Prova#

    • Dizer que SQL é procedural por natureza → errado; é majoritariamente declarativa.
    • Dizer que NULL = NULL é verdadeiro → errado.
    • Dizer que WHERE filtra grupos → errado.
    • Dizer que COUNT(coluna) conta todos os registros → errado se houver NULL.
    • Dizer que PRIMARY KEY aceita NULL → errado.
    • Dizer que LEFT JOIN retorna apenas correspondências → errado.
    • Dizer que DROP apaga apenas dados → errado; remove a estrutura.
    • Dizer que TRUNCATE é equivalente a DELETE ... WHERE condição → errado.
    • Termos absolutos como sempre, nunca, exclusivamente costumam invalidar itens.

    8. Simulado (Certo ou Errado)#

    1. SQL é uma linguagem exclusivamente procedural, exigindo que o usuário informe detalhadamente o plano de execução da consulta.
    2. O comando HAVING é utilizado para filtrar grupos resultantes de uma operação de agregação.
    3. Em SQL, a expressão coluna = NULL é a forma correta de testar valores nulos.
    4. Uma chave primária pode ser composta por mais de uma coluna.
    5. O comando COUNT(coluna) contabiliza todas as linhas da tabela, inclusive aquelas em que a coluna possui valor NULL.
    6. Em um LEFT JOIN, todas as linhas da tabela à esquerda são preservadas, mesmo sem correspondência na tabela à direita.
    7. O comando DROP TABLE remove a tabela e também sua estrutura do banco de dados.
    8. O WHERE é processado logicamente após o GROUP BY.

    Gabarito comentado#

    1. Errado. SQL é predominantemente declarativa. A pegadinha está em “exclusivamente procedural”.

    2. Certo. HAVING filtra grupos, normalmente após GROUP BY.

    3. Errado. O correto é IS NULL ou IS NOT NULL. Comparação com = não funciona para NULL.

    4. Certo. Chave primária pode ser simples ou composta. A tabela continua tendo uma única PK, ainda que formada por várias colunas.

    5. Errado. COUNT(coluna) ignora valores NULL. Quem conta todas as linhas é COUNT(*).

    6. Certo. Essa é a definição de LEFT JOIN. Linhas sem correspondência recebem NULL nas colunas da direita.

    7. Certo. DROP TABLE remove o objeto tabela, não apenas seus dados.

    8. Errado. Na ordem lógica, WHERE ocorre antes de GROUP BY. A pegadinha é confundir ordem escrita com ordem de processamento.
